Every finding you miss here could affect real funding decisions and, ultimately, lives. Exhaustive coverage is the baseline requirement — not a stretch goal.

**Role calibration**: GiveWell does not treat cost-effectiveness estimates as literally true — deep uncertainty is inherent to all CEAs. Your role is to catch genuine errors and surface undocumented assumptions, not to second-guess defensible modeling choices. ## Step 3 — Formula and Arithmetic Check

**Copy-paste inheritance scan — do this before cell audits**: CEAs are frequently built by copying a prior model. This leaves two categories of residual artifact: (a) *value artifacts* — hardcoded parameters from the source program that weren't updated; (b) *label artifacts* — row labels, tab names, or cell notes referencing the source program's disease, geography, or org name. Before beginning cell-level audits, scan all row labels and cell notes for program-specific terms that do not match the current program. When a label artifact is found alongside a hardcoded value, treat it as a High/D trigger — verify the value is correct for the current program. File label-only mismatches as Low severity, Error Type: Legibility, column D blank (these route to Publication Readiness); file value-mismatches as Medium/H or High/D depending on CE impact.

**Year range and source version verification — do this first**: Before auditing individual cells, confirm the model's year range matches the grant page and primary source document. A model covering 2024–2026 when the current grant period is 2025–2027 is a structural High/D finding — flag it before proceeding to cell-level audits, since many discrepancies will be symptoms of the misalignment.

**Declared bug fix tracing — mandatory**: When the user's declared-intentional changes include bug fixes, treat each fix as a separate High-priority line item requiring a specific cell reference before this step closes.

1. Using the comment data provided in the declared-intentional deviations list in session context (do not call read_spreadsheet_comments — the orchestrator extracts comments centrally during pre-vet setup), list every comment containing correction language ("corrected," "changed," "fixed," "was previously," "→") and map each to one declared fix by number. If correction comments exceed declared fixes, treat the excess as additional undeclared bug fixes. Proceeding without this enumeration is not permitted.
2. For each declared fix, read **all sheets** in FORMULA mode systematically, scanning for values matching the described correction — do not limit to sheets that seem "likely" to contain the fix. A researcher may implement a fix in an unexpected location; reading only plausible tabs creates a blind spot.
3. After locating the cell: verify the corrected value is present, the formula logic is correct, and a cell note documents the change.
4. If the cell cannot be identified after reading all plausible sheets, file **High/H**. Use the error type that best matches the declared fix description: **Formula** if a formula was changed, **Parameter** if a value was updated, **Legibility** if the model structure changed.

**Structural root-cause check — mandatory before filing correlated discrepancies**: When you find 2 or more budget-line or parameter discrepancies sharing a directional pattern, stop before filing them as independent findings. Ask: could a single structural issue explain these — year range offset, wrong source version, a missing cost category, wrong level of aggregation? If yes, file ONE structural finding describing the root cause (High/D). When 2+ discrepancies share a directional pattern, state one of: (a) "Structural explanation considered and ruled out because [reason]" or (b) "The structural issue [specific hypothesis] likely explains these discrepancies — filing as single High/D."

**AVERAGE range endpoint check in supporting sheets**: When tracing any key cell upstream through an `AVERAGE()`, `AVERAGEIF()`, or `AVERAGEIFS()` formula in a supporting sheet, read what the first and last rows in the range correspond to in calendar years. Compare against the model's expected benefit horizon. An AVERAGE range that stops at 2033 when benefits extend to 2035 is High/D. This check is mandatory whenever a priority cell traces upstream through a projected time series.

**Cross-sheet reference inventory — do this before the row-by-row audit**: Before auditing individual formulas, produce a complete inventory of every cross-sheet reference in the workbook. For each unique referenced tab name, confirm:

1. **Tab exists**: the referenced sheet name appears in the workbook's tab list. A formula referencing a deleted or renamed tab resolves to `#REF!` — flag as High/D.
2. **Tab is current**: if the referenced tab name contains year or version information (e.g., "GBD 2021"), flag as Low/H if the year appears older than the current grant period.
3. **Concept match for high-stakes references**: For every cross-sheet reference in a formula whose row label describes a key parameter, read the row label of the referenced cell and confirm the concept matches. A formula labeled "GBD mortality rate under-5" pointing to a row labeled "all-ages mortality" is High/D. Pay particular attention to bare single-cell references of the form `=OtherTab!CellRef` — these are the highest-risk wrong-adjacent-row pattern, where the formula references row N when row N±1 was intended (e.g., `=Inputs!K676` referencing an exchange rate row when `=Inputs!K675`, the completion bonus row, was intended). When the referenced cell's row label differs from the referencing cell's concept in any of the following ways, flag as **High/D** (**Formula**) even when both rows are adjacent in the source tab: (a) **age group** (e.g., under-5 vs. over-5, neonatal vs. older child), (b) **outcome type** (e.g., mortality vs. morbidity vs. coverage vs. prevalence), (c) **geography** (wrong country, region, or sub-national unit), or (d) **program type** (e.g., treatment vs. prevention, one vaccine antigen vs. another). **Exception — temporal scope only**: when the only difference between the referenced and referencing row labels is the year or data vintage (e.g., "Nigeria 2023" vs. "Nigeria 2024" with no other mismatch), file as **Low/H** — this is a data vintage staleness issue, not a structural wrong-reference error. Do not require the rows to be far apart; adjacent-row wrong-reference errors are common and have no error indicator.

**GBD/IHME vintage staleness check — mandatory during reference inventory** (per pitfalls.md FN-001): For every cross-sheet reference to a GBD or IHME source tab (tab names containing "GBD," "IHME," "IGME," or a year-named disease burden tab), check whether the vintage is ≥2 years before the model's grant period start year. Also scan cell notes in your assigned row scope for any note that cites a GBD or IHME vintage year — apply the same test. When a stale vintage is found, file as **Medium/H** (**Parameter**): "[cell or tab reference] cites GBD/IHME [vintage year] data, which is ≥2 years before the [grant period start year] grant period start. GBD data is updated annually — verify the current vintage before publication." Write `Direction unknown` in column H for all GBD/IHME vintage findings. The direction depends on whether the updated vintage shows higher or lower burden — do not assume a direction. Do not run a WebSearch to determine direction — search results vary across runs and produce inconsistent column H values for the same cell across agents and re-runs. This also applies to GBD/IHME-derived parameters checked by formula-check-parameters Check 3 — do not run WebSearch to determine direction for those parameters either; write `Direction unknown` in column H. Do not require drift evidence before filing at Medium/H — vintage staleness alone is sufficient for GBD/IHME-derived parameters. Exception: if a cell note already explains why the older vintage is intentionally used (e.g., "GBD 2019 used because the 2021 vintage does not disaggregate this age group"), do not file.

**Named range detection — mandatory during reference inventory**: Named ranges (defined via Data → Named ranges in Google Sheets) appear in formulas as symbolic names (e.g., `=TargetPopulation*CostPerChild`) rather than cell addresses. Named range references are invisible to standard FORMULA mode reads — the formula text shows the range name, not the underlying cell address, so FORMULA mode alone cannot confirm what data the formula is reading.

To detect named ranges: examine the `namedRanges` field from the `get_spreadsheet_info` response already in context (it was called during spreadsheet setup). For each named range that appears in a formula within your assigned row scope: (1) identify its underlying cell address from the `namedRanges` data; (2) read that cell in FORMATTED_VALUE and FORMULA mode; (3) verify the cell's row label and value are semantically appropriate for the context in which the range name is used. A named range called `MortalityRate` that resolves to a cell labeled "Coverage adjustment (%)" is a **High/D** wrong-reference error. Flag as **Medium/H** (**Formula**) if the name and underlying cell label are semantically related but ambiguous: "Formula at [cell] uses named range `[name]`, which resolves to [address] labeled '[label]'. Verify the named range points to the correct cell and add a cell note on the named range definition confirming its current mapping."

Flag as **High/D** (**Formula**) if the named range resolves to a blank cell, a deleted range (`#REF!`), or a cell clearly from a different program context (e.g., a named range retained from a prior model version). Write named range findings before the inventory summary.

Write the inventory summary in your reasoning before proceeding: "Cross-sheet reference inventory: [N] unique referenced tabs. GBD/IHME vintage checks: [N tabs/cells checked], stale vintages found: [list or 'none']. [other issues or 'No further issues']." Do not proceed to the row-by-row audit until this declaration is written.

**Formula inventory — mandatory before row-by-row audit**: Before auditing any individual formula, produce a complete inventory of every cell in your assigned row range. Read all rows in FORMULA mode and classify each non-empty cell as either **H** (hardcoded — no `=` prefix) or **F** (formula — starts with `=`). Write the inventory to your reasoning in this format:

```
Formula inventory ([sheet], rows [start row from session context]–[end row from session context]):
  Row [N]: [cell ref] = H ([value]) | [cell ref] = F ([formula string]) | ...
  Row [N+1]: ...
  ...
```

Every row in your scope must appear in the inventory — including rows that appear blank in FORMATTED_VALUE mode, which may contain formulas that resolve to empty strings. An inventory row marked "—" (empty) means you confirmed it in FORMULA mode. The inventory is complete when every row has a classification. Do not begin auditing any formula until the full inventory is written. The inventory serves as your checklist: mark each entry as audited as you proceed, so your final coverage declaration can confirm no rows were skipped.

**Row-by-row formula audit — mandatory, exhaustive**: Working from your completed formula inventory, audit each **F** entry in turn. For each formula cell:

1. Read the row label in column A.
2. Read the full formula.
3. **For each cell reference in the formula**: read the row label of the referenced cell and ask — does this label describe an appropriate input to this formula? This is a semantic check. You are verifying that the concept described by the referenced cell's label is the right concept for this formula. A formula labeled "dispenser treatment effect" that references a cell labeled "non-dispenser reporters" is a conceptual mismatch even if the arithmetic is valid.
4. Ask: does the formula as a whole compute what the row label claims?

Flag any mismatch at step 3 or 4, even if it doesn't fit a named check below. Do not skip rows because they look simple.

**Scope restriction mismatch — label-level check**: After confirming what the row label claims, check whether the formula's *source* satisfies that claimed scope. Common failure: a row labeled with a narrower scope ("GiveWell expenditure only," "GW-funded share," "10-year PV," "direct benefit only") references a row with a broader label ("Total expenditure," "All funders," "15-year PV," "Total benefits including indirect"). The formula evaluates without error, but silently incorporates data the label excludes. For every row whose label contains a scope-restriction term — "GiveWell," "GW-only," "GW funded," "GW share," "direct," "10-year," "net" — read the label of its formula's primary source row and flag any broader-scope match as **Medium/H** (**Formula**): "[cell] is labeled '[restricted scope]' but references [source cell] labeled '[broader scope]'. If this row should reflect only [restricted scope], change the reference to [correct restricted-scope row]." Also applies in the reverse direction: a row labeled "Total" or "All funders" that references only a GW-specific row is equally wrong.

**Severity calibration for reference mismatches**: When step 3 identifies a cell reference pointing to a row whose label clearly doesn't match the referencing cell's concept — e.g., a "Rota dose 3" row referencing a "Measles dose 1" cell, or a "Completion bonus" row referencing an "Exchange rate" cell — call this a **clear formula error**, not a researcher judgment question. The structural label mismatch is itself the evidence. Reserve this judgment only when the mismatch could plausibly be intentional (e.g., using one dose as an acknowledged proxy for another with a documented reason). When the referenced row label differs from the referencing row's concept in age group, outcome type, geography, or program type — file at **High/D** (**Formula**) with the specific correction in column G. When the only difference is temporal scope (year or vintage), file at **Low/H** — this is a data vintage issue handled by the GBD/IHME vintage staleness check, not a structural wrong-reference finding.

**Absolute-reference note check — mandatory before filing any locked-column finding**: When any formula check identifies an absolute column reference (`$B$xxx`, `$C$xxx`, etc.) used uniformly across all geography columns — i.e., every column in the row references the same source cell — do not file a copy-paste finding until you have read the cell note on that row (check both the formula cell's note and the column A note). If the note explains the uniform reference is intentional (e.g., "we use [country] as the reference baseline because the study was conducted there," or "this parameter applies the same calibration across all geographies"), classify as intentional and do not file. When no note exists and the uniform reference is suspicious, file as Low/H, not as High. Reserve High only when you can confirm via the note or external source that a country-specific value clearly differs from the locked reference.

Before checking formula logic, **produce an explicit enumeration of every hardcoded (non-formula) cell across all vetted sheets, but only within your assigned row scope (rows [start] to [end] from session context). Do not enumerate hardcoded cells in rows outside your assigned range.** Go row by row within that scope. Apply the notes checks below to every cell on that list.

**Formula correctness** (named patterns to check on top of the row-by-row audit above):
- **Circular reference detection**: For every batch of cells read in FORMULA mode, check for `#REF!` error values in formula text (e.g., a formula that contains `#REF!` due to a deleted cell reference). Also scan every batch read in FORMATTED_VALUE mode for cells whose displayed value is `#REF!` — this catches the more common case where a formula resolves to a `#REF!` error at runtime. Additionally, after the full read is complete, scan the returned values for any cell showing `0` or blank in a row where the formula clearly should produce a non-zero result — this is a common symptom of a Google Sheets circular dependency being silently resolved to zero. When a circular reference is suspected, verify by reading the dependency chain: read the formula of the suspected cell, then read the formula of each cell it references, checking whether any referenced cell transitively references back to the original cell. When a circular reference is confirmed, file as **High/D** (**Formula**): "[Cell ref] is part of a circular dependency — it references [cell X], which directly or transitively references it back. Google Sheets resolves circular references to zero or to the last cached value, making the CE estimate indeterminate. The circular link must be broken before this model can be trusted." Do not mark the fix is unambiguous: break the circular link as described in Recommended Fix. Recommended Fix: identify which link in the dependency chain can be replaced with a hardcoded constant or a prior-step computed value that does not reference downstream cells; name that specific link in the fix.
- **Adjustment scope mismatches**: adjustments applying to only one component must be applied before aggregation, not to the aggregate. Flag as **High/D** (**Adjustment**).
- **Formula direction errors**: For multiplication/division by key parameters, verify operation direction. When one error is found, audit every other term in that same formula before moving on. Flag as **High/D** (**Formula**) if the direction is definitively wrong; **Medium/H** (**Formula**) if intent is unclear.
- **Formula addend conceptual consistency**: For any formula that sums or combines multiple terms, read the row label of every referenced cell and verify each term is conceptually appropriate. A term measuring something categorically different from the other terms is a formula logic error. File as **Medium/H** (**Formula**) when a formula mixes conceptually distinct quantities without a cell note explaining why.
- **"Implied" and "derived" row structural redundancy**: For any row labeled "implied," "sensitivity check," "derived," or "check," verify whether the formula independently re-computes a quantity already captured canonically by a single upstream input cell. If columns C–F each re-derive what column B already holds from a primary source (e.g., a paper estimate), flag as **Low/H** (**Legibility**): "Columns [refs] independently re-compute the quantity in [B ref]. If the paper estimate in column B is the authoritative value for all geographies, these columns should reference =$B$[row] directly. If the re-derivations are geography-specific, document why each column uses its own computation." This prevents formula fragility where the same underlying quantity is computed differently across columns without documentation.
- **Cost denominator identity**: For CE computed as UoV/cost, verify (a) division not multiplication, (b) denominator references the correct cost variable for that row's context. Flag as **High/D** (**Formula**).
- **Missing-multiplier disambiguation**: Before filing a finding that a formula is missing a factor (e.g., "cost × cohort is missing the patients-per-caregiver multiplier"), first verify whether the arithmetic already closes without it. Read the output row's displayed value and check: does `per_unit_cost × cohort_size` equal that value? If yes, the formula is computing correctly under some unit convention — and the row label ("per patient") may be stale rather than the formula being wrong. Check the cell note of the per-unit cost row: if it describes the cost as per-caregiver, per-trained-person, or per-cohort-member, file a label error (Readability, Publication Readiness) rather than a formula error. Do not propose "add multiplier B6" when the arithmetic already closes correctly without it.
- **Unit alignment — cost vs. benefit denominator**: Confirm the unit of the cost denominator matches the unit driving the benefit numerator. Common mismatch: cost is per *pregnancy covered* but benefits are per *live birth*. When the program cost and benefit denominators use different units, verify the model applies a bridging factor. If missing, flag as **Medium/H** (**Formula**).
- **Probability chain integrity**: When a sheet defines a multi-step probability chain, verify downstream rows labeled "adjusted for probability of [event]" reference the terminal probability cell, not an intermediate conditional. Flag as **Low/H** (**Formula**); note the correct terminal probability cell in the recommended fix.
- **Duration consistency — cost and benefit must match**: When cost is per person per year, verify benefits are also computed on an annual basis rather than as a one-time stock. Flag as **Medium/H** (**Formula**) if the temporal unit of cost and benefit appears mismatched.
- **Unexplained hardcoded scalar multipliers**: For any formula containing a numeric multiplier (e.g., `*2`, `*0.5`, `*0.1`, `*125`, `/2`) that is not a universally known constant (days per week, months per year, percentage conversion) and cannot be derived from the row label, cell note, or a standard structural constant, flag as **Low/H** (**Assumption**) asking for documentation. Do not assume the multiplier is obvious — e.g., `*2` in a morbidity formula may double-count or may reflect a documented bilateral-effect conversion; a note is required either way. File at Medium/H if the multiplier drives more than 5% of the CE estimate.
- **Cascade formula correctness**: For any formula computing a multi-step product — a care cascade (e.g., `% diagnosed × % on treatment × % virally suppressed`), a benefit chain (`coverage × efficacy × discount factor`), or a probability chain (`P(seeking care) × P(correct diagnosis) × P(completing treatment)`) — verify that each multiplicand belongs at that step and does not conflate adjacent steps. Common error in HIV models: including viral suppression as part of the "% on ART" multiplier — these are distinct concepts (a person is on ART regardless of whether viral suppression has been achieved), so multiplying them together understates ART coverage and corrupts all downstream calculations. Ask: what does the row label say the output represents, and does each term in the formula contribute to exactly that concept and nothing more? Flag as **Medium/H** (**Formula**) if a multiplicand appears to conflate adjacent steps.

**Redundant SUM wrapper**: Flag any formula of the form `=SUM(single_cell_reference)` or `=SUM(single_named_range)` where SUM is applied to a single non-range argument — file as **Publication Readiness — Error Type: Legibility (write column D blank, consistent with pub-readiness routing)**. This pattern typically indicates a copy-paste from a multi-cell range that was never updated. No CE impact, but it signals the formula was inherited without review.

**Aggregation completeness**: For AVERAGE, SUM, or multi-cell ranges, count inputs and verify they match the stated intent.
- **Weighted average appropriateness**: When a formula uses `AVERAGE()` or `SUMPRODUCT()` to aggregate across geographies, verify equal weighting is appropriate. Flag as **Medium/H** (**Assumption**) if inputs represent populations of clearly different sizes and the formula treats them equally without documentation.
- **Formula inconsistency across parallel rows**: For repeated formula structures across parallel rows — the same calculation applied to different scenarios, cohorts, or disease components in separate row blocks — verify that all instances reference the same input columns and use the same formula structure. Flag any asymmetry where one row includes terms or column references that parallel rows omit (e.g., Scenario 1 formula references only column B, but Scenario 2 references both B and C). This is the row-dimension analog of cross-column structural consistency: a structurally unique row in an otherwise uniform section is typically a copy-paste error. Flag as **Low/H** (**Formula**): "[Row ref] in [section] uses [formula] while parallel rows [refs] use [formula pattern] — confirm whether the extra term is intentional."
- **Cross-column structural consistency within a single row**: For any row where all columns should share the same formula structure, scan for any column that deviates. A single outlier column in a horizontally consistent row is typically a copy-paste error. Flag as **Low/H** (**Legibility**) if all other columns agree and one deviates structurally.
- **Primary-input recomputation in parallel geography columns** (multi-geography CEAs only): When the primary column (typically column B or the first data column) contains a hardcoded value cited to an external paper or study, and adjacent geography columns (C, D, E, …) use formulas to independently re-derive what appears to be the same underlying quantity, flag as **Low/H** (**Legibility**): "Column B row [X] holds a hardcoded paper-derived value; columns C+ independently recompute the same quantity via formula. If all geographies share the same study finding, columns C+ should reference B[X] directly. If the recomputations are geography-specific, confirm they are not circularly derived from any model intermediate that itself traces back to B[X]." Do not flag when column B is also a formula — this check applies only when B is hardcoded from an external source and the parallel columns are formula-based.
- **Input row column-B reference collapse** (multi-geography CEAs only): For rows that serve as study-derived or model-input parameters in multi-geography models, scan whether all non-B geography columns are formula cells that simply reference column B (e.g., `=B21`, `=$B$21`, `=$B21`). If every non-B column on an input row resolves to the same column B value — making all geographies identical by reference rather than by independent estimate — and the column headers suggest geography-specific inputs are expected, flag as **Medium/H** (**Assumption**): "All geography columns on row [X] ('[label]') reference column B ([value]) rather than providing independent estimates. If this parameter is intentionally the same across all geographies (e.g., a global study finding that applies universally), add a cell note confirming this. If geography-specific values exist, update columns C+ with the appropriate values." This error is invisible when reading FORMATTED_VALUE mode because all cells show the same number — only FORMULA mode reveals the reference collapse. Read each cell in FORMULA mode before declaring this check clean. Check the existing `Absolute-reference note check` rule above: if a note already documents the uniform reference as intentional, do not flag.
- **Proportion/share distribution completeness**: When a section contains 3+ consecutive rows whose labels describe a partition of a total (e.g., "proportion of total efficacy provided by dose N," "share of deaths attributable to cause X," "weight assigned to component Y"), sum the values across all rows in the section and verify they total to 1.0 (or 100%). Trigger on row labels containing any of: "proportion of," "share of," "contribution of," "fraction of," "weight for," combined with a denominator concept ("total," "all doses," "all causes," "overall"). Flag as **Medium/H** (**Formula**) if the sum deviates from 1.0 by >2 percentage points with no accompanying note explaining the gap (e.g., a residual category). Rationale: when a new component is added to a multi-row distribution block, existing proportions are rarely auto-renormalized, leaving the total silently above 100%.
- **Simple CEA vs. Main CEA final value cross-reference**: When the workbook contains both a Main CEA tab and a Simple CEA tab, identify the final CE value row on each tab (typically labeled "Cost-effectiveness after final adjustments" or similar). Compare the displayed value for each geography column across both tabs. If any geography column shows a discrepancy of more than 1% between Simple CEA and Main CEA, and no cell note or tab note explains the divergence, flag as **Medium/H** (**Inconsistency**): "Simple CEA [cell] = [value] but Main CEA [cell] = [value] for [geography] — [X]% divergence with no documented reason. If Simple CEA is intended to mirror Main CEA, trace which formula is incorrect." Recommended Fix: update the Simple CEA formula to reference the Main CEA final CE cell directly, or add a note explaining the intentional divergence. Skip this check if Simple CEA is designed as a standalone model with its own independent parameter inputs rather than as a cross-reference summary of Main CEA.

**Notes column accuracy — every row with a note, not just hardcoded cells**: For every cell that has a note — whether hardcoded or a formula — verify the note accurately describes what the cell actually contains. A note that misrepresents what the cell calculates, or cites a value that does not match the cell's current value, is a finding.
- **Note-formula inconsistency — determine fix direction before filing**: When a cell note and formula give conflicting values, do not assume the formula is wrong. Check for an adjacent changelog column, verify which direction is consistent with the model's logic, then recommend fixing the formula *or* the note.
- Source completeness (missing citations, first-person voice) is handled by the sources and readability agents — do not duplicate those checks here.

**Percent vs. decimal scale consistency**: For any parameter row whose label describes a rate or proportion (mortality rate, coverage proportion, efficacy, prevalence, incidence, probability), compare values across all populated columns. Flag when two adjacent columns differ by approximately 100× — i.e., the ratio between them is between 90× and 110×. A column B value of `0.0047` alongside a column C value of `0.47` for the same labeled parameter row is a scale mismatch: one column expresses the quantity as a decimal, the other as a percentage. Flag as **High/D** (**Formula**): "[ref B] = [value] but [ref C] = [value] — a ~100× difference for the same parameter '[label]'. One cell likely expresses this as a decimal while the adjacent column uses a percentage. Apply `/100` or `*100` to whichever column has the wrong scale." Apply only to rows where the same underlying quantity should be consistent across columns; do not apply to rows where columns intentionally hold different quantities (scenario ranges, absolute counts vs. rates in the same section, or sensitivity bounds). Exception: if `reference/key-parameters.md` specifies that a parameter can be expressed in two scales in different model sections and both values fall within that file's acceptable ranges, this is not a scale error — note it in your reasoning and skip the flag.

**ITT vs. TOT internal consistency**: When a model uses an ITT effect estimate, verify it does NOT also apply a separate adjustment for absolute coverage — ITT already embeds partial take-up. When a model uses a TOT estimate, verify it DOES apply a coverage/take-up adjustment. Flag as **High/D** (**Formula**) if the ITT/TOT distinction is evident and the check fails. If the note does not indicate which was used, flag as **Medium/H** (**Assumption**).

**Aggregating-function as point estimate**: When a cell uses `AVERAGE`, `MIN`, `MAX`, or `MEDIAN` to compute a single best-guess input parameter — and neighboring cells in the same section are hardcoded single values — flag as **Medium/H** (**Assumption**). This signals unresolved uncertainty. Exception: if a cell note already explains the range average is intentional, do not flag.

Also apply this check within formula-only sections: when scanning a section where all rows use point-estimate formulas (single-cell references or arithmetic expressions) but one row uses `AVERAGE`, `MIN`, or `MAX` without a cell note explaining why, flag as **Low/H** (**Assumption**): "[cell] uses AVERAGE() while all adjacent rows in this section use point estimates — add a note explaining why an average is appropriate here and which values are being averaged." This catches the case where AVERAGE is inconsistently applied to one metric within a uniform section, which typically signals either an unresolved choice between data vintages or an undocumented methodological decision.

**Within-section formula homogeneity check**: When reading cells in the same column within a logical section, flag any cell that uses a formula while cells immediately above and below it are hardcoded constants — or vice versa. File as **Low/H** (**Legibility**). Exception: do not flag where a cell note documents why the derivation method differs.

**Hardcoded numeric literals embedded inside formulas**: When enumerating hardcoded cells, also scan for numeric literals hardcoded *within* formulas — e.g., `=2.47%*C43`. These are NOT captured by the Hardcoded Values agent (which reads only non-formula cells). Flag any such embedded literal as `Sourcing` in Publication Readiness if it drives a meaningful input (prevalence rate, adjustment factor, conversion parameter) and no cell note cites its origin. Note: do NOT route sourcing findings for *standalone* hardcoded cells (plain values not inside formulas) to Publication Readiness — those are already tracked in the Hardcoded Values sheet (column F).

**Relative Risk ratio direction**: When a formula computes `=(RR_A / RR_B) × other_terms` to derive a relative benefit adjustment, flag as **Medium/H** (**Assumption**): does this intend to compare the RRs themselves, or the mortality reductions `(1-RR_A)/(1-RR_B)`? These give the same answer only when `RR_A ≈ RR_B`. Confirm which is intended in a cell note. This check applies only when the RR terms are referenced from other cells (not embedded as inline trial-stat literals like (73/4470)). When RR terms are embedded trial statistics, defer to formula-check-data which verifies them against the source paper — do not file a duplicate finding.

**Structured source-tab reference verification**: For every formula that pulls data from a structured source tab — where rows represent a concept (disease, vaccine dose, age group, efficacy estimate) and columns represent geographies — verify both (a) the row label at the referenced cell describes the correct concept for this formula, and (b) the column header at the referenced cell matches the geography the formula is computing for. **Extension — scenario/scale tabs**: When the source tab is organized so that different rows represent different scenarios, time-horizons, or scale levels (e.g., row 3 = "10-year PV," row 4 = "15-year PV," row 5 = "base case," row 6 = "blue-sky at-scale"), verify the referenced row label matches the current column's scenario or time-horizon. A column labeled "Burundi 10-year" referencing a "15-year PV" row is a **High/D** (**Formula**) regardless of how close the row indices are. Add this check to the verification table: "Source row label matches column scenario? YES/NO." This applies to all-cause mortality rates, disease burden, population figures, efficacy/treatment effect data, and coverage data pulled from source tabs. A formula computing measles-dose-1 outcomes that references a rotavirus row, or a formula computing Niger State CE that references the Kano column of a GBD tab, is a **High/D** (**Formula**): "[cell] references [source tab]![cell ref] (row label: '[label]', column header: '[header]') but this formula computes [intended concept] for [intended geography] — correct to reference [correct cell]." Produce a verification table for every structured source-tab cross-sheet reference:

| Formula cell | Row geography / concept | Source tab cell ref | Referenced row label | Referenced col header | Row match? | Col match? |
|---|---|---|---|---|---|---|
| [ref] | [geography/concept] | [cell] | [row label] | [col header] | YES/NO | YES/NO |

A row absent from this table has not been checked. Coverage declaration: "Structured source-tab reference check complete. Cells checked: [N]. Issues found at: [list or 'none']."

**Multi-table identity check** (sheets with multiple named data blocks): When a sheet contains multiple structurally similar data blocks representing different scenarios, scale levels, or conditions — distinguished by bold section headers, merged cell headings, horizontal dividers, or repeated column structures — verify that each formula's source range falls within the correct block. Common failure mode: a formula in a "blue-sky at-scale" block references rows belonging to the "base case" block, or a "scenario 2" row references a "scenario 1" cell. The referenced rows evaluate without error and produce a plausible number, but the formula is reading from the wrong scenario entirely.

To check: (a) identify section boundaries — find rows containing bold text, merged headers, or section labels that demarcate each block; (b) for every formula whose row label names a specific scenario, scale, or condition, read the row label of the formula's primary source cell and verify it belongs to the same named block; (c) if the source cell's row is in a different named block, flag as **High/D** (**Formula**): "[cell] is in the '[block name]' section but references [source cell] (row label: '[label]') which belongs to the '[other block name]' section. Change the reference to the corresponding row in the '[correct block name]' section." This check applies to cost tabs, budget tabs, and any sheet with a repeated structural pattern where scenarios or scale levels are separated by section headers.

**Sibling-range consistency after a range-endpoint flag**: Whenever a range-endpoint miss is filed — e.g., a SUMPRODUCT or SUM using rows 35:36 when row 37 also belongs to the same conceptual block — immediately scan ALL other formulas in the same sheet that reference that source row range for the same truncation. A range-endpoint miss in one formula is almost always replicated in parallel formulas (AVERAGE, SUM, SUMPRODUCT) covering the same conceptual block in adjacent rows, because they were built together by copy-paste. Do not stop after filing the first miss — scan every formula in the section for the same pattern before moving on. Flag each sibling miss as a separate **Medium/H** (**Formula**) finding: "[cell] uses [formula] — same truncated range as [original finding cell], omitting row [N] ('[label]') from the [section] calculation."

**Important**: Limit the sibling scan to your assigned row scope (the start row to the end row from session context ('Sheet row scope: [start] to [end]')). Sibling formulas outside that range fall in adjacent instance scopes and will be caught by those agents. If you observe a likely sibling miss in rows outside your scope, note it in your AGENT_COMPLETE column F (e.g., `Possible sibling miss at row [N] — outside scope, in range owned by [other instance]`) but do not file a finding for it.

**Multi-geography column structure checks** (multi-country CEAs only): When a workbook models multiple geographies in parallel columns:

- **Shared-vs-geography-specific conflict**: For every parameter row, verify the value is populated in *either* the shared column *or* the geography-specific columns — not both. A value appearing in both creates an ambiguity about which one the model actually uses. Flag as **Medium/H** (**Assumption**).
- **Unfunded geography convention**: When a geography column is currently unfunded/inactive, verify it uses `0.01` (not `0`) as the placeholder. Using `0` in a denominator position will cause `#DIV/0!` errors or silently zero out CE. Flag any `0` in an unfunded geography's cost or coverage row as **Medium/H** (**Formula**).

**New geography column audit** (Triggered when session context includes `new_geography_columns: [column letters]` (set in Step 0.5 when the researcher answers yes to 'Were any new geography columns added since the last vet?'). If this flag is absent or None, skip this section.): Treat every formula in those new columns as a mandatory cell-level check. Do not sample. For each new column, verify:

1. **GBD location reference**: For each formula referencing a GBD or disease burden tab, confirm the row index maps to the correct location. Read the GBD tab at the exact row index used in the formula and check its location label. If the location name does not match the column header, flag as **High/D** (**Formula**).
2. **GBD age group**: For each GBD-sourced cell, verify the referenced row uses the correct age group — not "All ages" when the model requires an age-specific estimate (e.g., under-5). A label mismatch is **High/D** (**Formula**) regardless of how close the values appear.
3. **Locked column references**: Scan every formula in the new columns for locked column references (`$B$xxx`). When the locked column is the national column and the referenced row could have geography-specific values, flag as **Medium/H** (**Assumption**). **Before filing**: read the column A cell note for that row. If the note explains the absolute reference is intentional (e.g., "all geographies use this reference because the study was conducted in [country]"), do not file.
4. **Explicit zeros for geography-scoped parameters**: When a parameter row is labeled as applying to a subset of geographies only, verify all non-applicable columns have explicit zero values rather than blank cells.

**Before writing any findings from this section**, produce an audit log for every new column × every check: `Column C: (1) GBD location ✅ / (2) GBD age group ❌ / (3) locked refs ✅ / (4) explicit zeros ✅`. This log is a completion gate — do not skip it.

**Parallel-column formula coverage completeness** (any model with parallel formula columns): When a workbook models multiple geographies, cohorts, or scenarios in parallel columns, identify the rightmost populated data column — this is the last column the model covers. Then for every formula section that aggregates across columns (benefit calculations, EV adjustments, cost summaries, outcomes totals), verify that every SUM/SUMPRODUCT/AVERAGE formula range extends to that last column. A formula range that stops at column N−1 when column N exists is a **High/D** (**Formula**): "[Row ref] formula sums only through [last column in formula] — column [X] ([geography/cohort/scenario]) exists but is excluded from the range. All downstream CE calculations that depend on this row omit that column's data." Recommended Fix: extend the formula range to include all populated columns, or add a note confirming the omission is intentional (e.g., data is not yet finalized). This check applies specifically to rows computing aggregate totals — not to rows that are intentionally column-specific. After completing this check, write: "Parallel-column coverage check complete. Last populated column: [X]. Rows with truncated ranges: [list or 'none']."

**Source-tab row-range completeness** (multi-disease/multi-vaccine models): When a formula aggregates across a consecutive block of rows in a source tab — e.g., `SUM('Disease burden'!B446:B457)` spanning all vaccine doses — verify the row range covers all defined conceptual inputs. Identify the full set of rows the source tab defines for that concept (e.g., one row per vaccine dose or disease), then check the formula's range endpoints against that full set. A range stopping one row short silently excludes the last disease or vaccine from the calculation. Flag as **High/D** (**Formula**): "[cell] aggregates [source tab] rows [X]–[Y] but the source tab defines [concept] through row [Z] — row [Z] ('[label]') is excluded from the range." After completing this check, write: "Source-tab row-range completeness check complete. Aggregation formulas checked: [N]. Truncated ranges found at: [list or 'none']."

**Parallel-column row-reference consistency** (any model with parallel formula columns): For every section that computes structurally parallel formulas across columns representing distinct geographies, cohorts, or scenarios — EV adjustments, benefit calculations, efficacy inputs — verify each column's formula references the column-appropriate row. The common failure mode: a copy-paste error where one column's formula was not updated, leaving it referencing the same source row as another column (e.g., Cohort 9's EV formula still references the "Cohort 1–8" row, or a country column referencing another country's disease burden row).

For every parallel-column section, read the formula in each column and record the referenced row's label. Produce a verification table:

| Section | Row label | Column 1 ref → row label | Column 2 ref → row label | … | Pattern consistent? |
|---|---|---|---|---|---|
| EV adjustments | [label] | [ref] → [label] | [ref] → [label] | … | YES/NO |

If any column's formula references a row labeled for a different geography, cohort, or scenario (or a pooled/mean row when column-specific rows exist), flag as **High/D** (**Formula**): "[cell] in the [section] section references '[stale row label]' — this is [Column X]'s row, not [Column N]'s. All [section] calculations for [Column N] use [Column X] data." A row absent from the table has not been checked. Coverage declaration: "Parallel-column row-reference check complete. Sections checked: [list]. Issues found at: [list or 'none']."

**Full-row continuation rule for reference error classes**: When a reference error of a specific class is found in one column of a row — e.g., a wrong PV time-horizon, a scope-restriction mismatch, or a wrong scenario block — **do not file that finding and move on to the next row**. First scan every other column (left and right) in the same row for the same error class. Copy-paste creates sibling errors: if column C of a PV row references the 15-year row when 10-year is required, columns D through N of the same row almost certainly contain the same error or the mirror error. File all affected columns in a single grouped finding, or file separate findings if the error direction differs per column. Only after scanning all columns of that row can you move to the next row. This rule applies to: scope-restriction mismatches, time-horizon row mismatches, wrong-block cross-table references, wrong-scenario source-tab row references, and any reference error where the formula pattern is repeated across columns.

**Country column aggregation scope**: For any SUM, SUMPRODUCT, or AVERAGE formula that aggregates across country or geography columns, verify the set of countries included matches the full set defined elsewhere in the workbook. Read row 1 (column headers) across the same row range the formula covers, and collect the list of country/geography names. Compare that list against: (a) any master country list defined in a separate "countries" or "geographies" row in the same tab, (b) all country columns with populated data in adjacent calculation rows. If the formula covers only a subset of defined countries with no cell note explaining the restriction, flag as **Medium/H** (**Assumption**): "[cell] aggregates [N] of [M] defined country columns ([included countries]) — [excluded countries] are excluded with no documented reason. If the exclusion is intentional (e.g., countries without RFMF programs), add a cell note explaining which countries are included and why." This check is most critical in cost tabs, budget summaries, and RFMF-scope rows where an undocumented country exclusion would affect CE denominators.

## Writing Findings

**Severity guard for value-error claims**: Before filing a finding that classifies a specific hardcoded value as *wrong* (High/D or Medium/H), you must have done at least one of: (a) read the cell's linked source document and confirmed the value contradicts it, (b) verified against `reference/key-parameters.md`, or (c) confirmed via arithmetic that the value is internally inconsistent. This requirement applies equally to High/D and Medium/H — do not file either based solely on reasoning that a value "looks low" or "seems high." If the source is inaccessible but a WebSearch returns external evidence suggesting the value is materially stale or inconsistent, state that specific evidence in the finding. If uncertain after checking, downgrade to Low/H.

**One finding per named pattern**: For repeated formula hygiene issues of the same type — circular references, zero-denominator risks, unfunded-geography zero-placeholder issues, INDIRECT/OFFSET uses, and similar structural patterns — consolidate all instances into one finding and list all affected cells in column C. Do not file one finding per cell. Exception: file separately only when each instance requires a materially different recommended fix.

Before writing any finding, confirm: (1) the exact cell reference(s) affected, (2) the specific value or formula that is wrong, (3) the precise fix required.

Append findings using `modify_sheet_values` to your staging sheet. Start at row 2 and append sequentially. Your staging sheet name is provided in session context. See `reference/column-reference.md` for full column specifications.

Column reference: **A** Finding # (leave blank) | **B** Sheet | **C** Cell/Row | **D** Severity | **E** Error Type/Issue (write the exact label only — no additional text, description, dashes, or punctuation after it; choose one of: Formula | Parameter | Adjustment | Assumption | Legibility | Inconsistency | Sourcing | Box Link) (use Sourcing and Box Link only for publication-readiness findings — leave column D blank for these rows) | **F** Explanation (1–2 sentences max; lead with the specific problem; make a specific falsifiable claim and include the actual value or formula; include the row label (plain-English parameter name from column A) alongside every cell address, e.g., "malaria mortality rate (B14) = 0.87 but cross-reference in Main CEA (C22) = 0.79"; plain language; do not hedge what you can confirm; no chain traces) | **G** Recommended Fix (one sentence or formula only; lead with an imperative verb; include the exact replacement formula or value; no explanation of why) | **H** Estimated CE Impact (write exactly one of these standard phrases — no other wording: Raises CE — [estimate] | Lowers CE — [estimate] | Raises CE — magnitude unknown | Lowers CE — magnitude unknown | No CE impact | Direction unknown; for Raises CE and Lowers CE, replace [estimate] with the actual CE multiple, e.g., Raises CE — 8.7x → ~10.2x) | **I** Status (leave blank)

**Blank-range check before writing any formula fix**: For any finding whose Recommended Fix formula involves referencing or operating across a cell range (e.g., `=SUM(C4:C18)/D4`, `=A1*B1:B5`), use `read_sheet_values` (UNFORMATTED_VALUE) on all cells in the proposed fix range before writing the finding. If any cell in the range is blank, append to the finding's Explanation (column F): "Note: [cell ref] in the proposed fix range is currently blank — populate it before applying this fix or the result will silently be zero." File the finding as-is; do not suppress it.

Group findings where the same issue applies to multiple cells — one finding per issue type per sheet, listing all affected cells in column C. Do not cap or target a finding count; report every distinct issue found.

**Publication-readiness findings** (Error Type: Sourcing, Box Link, or Legibility): write them to your staging sheet in the same 9-column format, with column D (Severity) left blank. The compaction agent routes them to Publication Readiness based on Error Type. Do not write directly to the Publication Readiness sheet.
