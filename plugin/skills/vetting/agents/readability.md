**Stakes — why this matters**: GiveWell allocates hundreds of millions of dollars in grants based on cost-effectiveness analyses like this one. A missed formula error, a stale parameter, or an uncaught copy-paste bug can cause CE estimates to be overstated by 2–10×, directing funding toward less effective interventions or away from more effective ones. Every finding you miss here could affect real funding decisions and, ultimately, lives. Exhaustive coverage is the baseline requirement — not a stretch goal. Exhaustion is not an excuse for stopping early. The Role calibration block below governs how to *classify* what you find — not how thoroughly to look for it. Thorough coverage and conservative severity are both required.

**Role calibration**: GiveWell does not treat cost-effectiveness estimates as literally true — deep uncertainty is inherent to all CEAs. Your role is to catch genuine errors and surface undocumented assumptions, not to second-guess defensible modeling choices. When a researcher's approach is reasonable but undocumented, prefer a clarifying question (Low) over a finding (Medium/High). Reserve Medium and High for factual errors, internal inconsistencies, or missing required elements.

**Coverage mandate — no shortcuts**: Every check in this file applies exhaustively to all rows, all columns, and all cell notes across every vetted sheet — no sampling, no section-restricted scope, no "key cells only." Read every row from first to last. Check every column with data. Read every cell note. Do not stop early when you have found several examples of an issue type — check every remaining row. Do not skip rows because they look similar to prior rows or because the sheet is small. **Finding the first instance of an issue type does not conclude that check** — continue checking all remaining rows and columns before writing findings. After completing each major check section, write two coverage declarations before moving on using the canonical format: (1) `COVERAGE | readability | [check name] | [rows X–Y checked] | issues found: [N] | status: complete` (2) `COVERAGE | readability | [check name] — notes | rows [X–Y] notes read: [N] | issues found: [N] | status: complete`. ## Step 7 — Readability and Label Review

Cross-check against the Guide to Making Spreadsheets Legible, then:

**⚠ DO THIS FIRST — Simple CEA section ordering (top-charity CEAs)**: Before any other check, locate any sheet or section labeled "Simple CEA" and verify the rows follow the calculation sequence: (1) coverage or delivery parameters, (2) effect size or lives-saved estimates, (3) cost inputs, (4) cost-per-outcome calculation, (5) CE multiple versus benchmark. Flag as **Medium/H** if the CE ratio or benchmark comparison row appears above the cost inputs or mortality estimates it depends on. This is the single most frequently missed readability finding across all vets — check it immediately, before reading any other row, so exhaustion cannot prevent it. The acceptable exception is a single top-summary row that mirrors a CE result computed in full below. File as Legibility. Write a coverage declaration here: "Simple CEA ordering checked: [found/not found]. Result: [issue or 'pass']." Do not proceed to other checks until this declaration is written.

**Label precision**: Read every row label and every column header as a first-time reader. Flag labels that are imprecise or potentially misleading, inconsistent with what the formula actually calculates, or referencing the wrong organization or context. For column headers specifically: scan row 1 and any other header rows for placeholder-sounding text ('Arbitrary value', 'Column X', 'TBD', 'N/A') or headers that don't accurately describe the data below them — column headers are reader-facing and stale or generic ones are a publication-readiness issue. Flag as Low/O.

**Wrong intervention vehicle or delivery mechanism in labels**: As part of label precision, verify that any food vehicle, drug, or delivery mechanism named in row labels (e.g., rice, wheat flour, oil, salt, bednet, tablet, injection) matches the program's actual intervention. Template errors — where a label names the source model's vehicle rather than the current program's — are common and publication-critical. Example: a label reading "consuming iron-fortified wheat flour per year" in a rice fortification model. Flag as Low/O with a recommended rename to the correct vehicle.

**Multi-scenario column header accuracy**: For CEA tabs with multiple scenario columns (e.g., "Current scale," "Proposed scale," "Ansh's estimate"), verify that each column header accurately describes both the treatment effect source and the counterfactual source used in that column. A mismatch — where the header implies one data source (e.g., "Ansh's estimated") but the formula for a key row (mortality counterfactual, effect size) pulls from a different source (e.g., GBD, Cochrane) — is a Low/H finding. The column header is the first thing a reader uses to interpret the entire column; a mislabeled header creates systematic misreading of every value in it. Check each scenario column's key rows (mortality baseline, effect size, IV, EV) against its header claim. Pay special attention to age-range labels: whenever a row label states an age range (e.g., "under age 5," "under-five," "children under 5"), verify it matches the formula's actual age group. VAS CEAs target 6–59 months post-neonatal; labels should read "aged 5–59 months" for the primary benefit stream, not "under age 5." Flag any mismatch as Low. Additional label patterns to flag: (a) **mechanism-generic labels** — a benefit-stream row labeled "Deaths under 5 averted" without identifying the causal mechanism when the workbook models a specific intervention (eLMIS, KMC, VAS, SMC); (b) **undescriptive transformation labels** — intermediate rows labeled "Adjusted value" or "Modified estimate" without specifying the adjustment ("after external validity adjustment," "discounted to present value"); (c) **scope-mismatched labels** — a label describing a broader scope than the formula covers (e.g., "All SSA countries" when the formula covers only two countries).

**Scenario naming cross-consistency**: When a workbook uses named scenarios (e.g., "Conservative," "Base," "Optimistic," "Scenario 1," "High-end"), scan all column headers, row labels, and cell notes across all vetted sheets to confirm names are used consistently. Flag as Low/O if the same scenario is called by different names in different places — e.g., "Conservative" in a column header but "Low" in a cell note referring to the same column, or "Scenario 2" in the Main CEA and "Upper bound" in the Simple CEA for the same set of inputs. Recommended fix: standardize to the most descriptive name and apply it everywhere.

**Unnecessary rows**: Flag any row whose output is not referenced by any downstream formula — may be vestigial and could confuse readers. To check this rigorously: using the FORMULA mode data, compile every cell reference that appears in any formula across the entire sheet. Make targeted `read_sheet_values` calls in FORMULA mode section by section — process each logical section (parameters, calculations, adjustments, outputs) in sequence rather than in a single full-sheet batch. Any row whose cells do not appear in that compiled reference list is a vestigial row — flag it as Low/O. Do not rely on visual inspection or row-label inference; trace references explicitly. **Before flagging a row as vestigial**: check whether it is consumed by an indirect lookup formula. A row may not appear in any direct cell reference (e.g., `=$E$47`) but still be consumed by `=INDEX($E$40:$E$50,MATCH(...))`, `=VLOOKUP(...)`, `=HLOOKUP(...)`, or `=OFFSET(...)` — in which case the row is active, not vestigial. Scan all formulas in the sheet for INDEX, MATCH, VLOOKUP, HLOOKUP, and OFFSET functions that reference ranges spanning the candidate row before filing. Common failure mode: intermediate calculation rows added during development are retained after their purpose is superseded; they still evaluate but nothing reads them. **File vestigial rows with a specific deletion recommendation** — not "may be vestigial." The recommended fix should read: "Delete row [N] before publication — output not referenced by any downstream formula." Also flag rows that appear to be earlier-version calculations left beside their replacement: duplicate row labels, adjacent rows computing the same quantity by different methods, or rows explicitly labeled "old," "previous," "v1," or "prior [year]" without a note explaining why the historical version is retained. Pre-publication cleanup misses are consistently among the most common O-recall gaps — treat this check as mandatory, not optional.

**Tab ordering**: Sheets should flow left (summary/output) → middle (calculations) → right (source/inputs) — the reader arrives at results and navigates right toward evidence. Flag as Low/O if source or raw-data tabs appear to the left of output or CEA tabs; flag as Medium/H if unrelated or vestigial tabs are interspersed among live calculation tabs without clear separation. Note: output-left is the GiveWell convention — a reader arriving at the workbook should see the bottom-line CE without scrolling right.

**Section ordering within tabs**: Within each sheet, the standard sequence is: parameters/inputs at the top, intermediate calculations in the middle, outputs and adjustments at the bottom. Flag as Low/O if cost inputs appear after the benefit calculations that depend on them, or if an adjustment row appears before the value it adjusts. Exception: a CE summary or bottom-line output row placed at the very top of a sheet for reader navigation is a common and acceptable GiveWell pattern — do not flag this. Only flag when an input or parameter row appears after a formula that depends on it, causing a reader to have to scroll backward to verify the logic.

**Simple CEA section ordering**: See the mandatory first check at the top of Step 7 — do not re-run here.

**Calculation flow and ordering**: Flag when rows are not in logical sequence — e.g., a parameter used before it is defined, or adjustments appearing before the values they modify. To check this explicitly: for any non-trivial formula result row, verify that all cells it depends on have lower row numbers in the same sheet. When a derived value appears *before* one or more of its inputs in the same sheet, flag as Medium/H — this forces a reader to scan out of order to verify the logic. Recommend moving the dependent row below its inputs, or adding a note explaining the intentional ordering.

**Reader walkthrough**: Step through the sheet top-to-bottom as a first-time reader. At each row: would I understand what this is doing without opening the cell? Is the calculation flow intuitive? Flag anything that caused you to pause, even if technically correct.

**Notes column completeness**: This check is handled by the dedicated notes-scan agent (Step 7c) running in parallel — do not duplicate it here. Step 7c covers: missing "Calculation." notes on formula rows, missing source annotations on hardcoded rows, template boilerplate, raw URL notes, first-person language, and row label quality (redundant, directionally misleading, and scope-mismatched labels). You will see those findings in the Publication Readiness sheet.

**Number formatting vs. row unit**: When reading FORMATTED_VALUE data, scan every row in the entire sheet for a mismatch between the number format applied to value cells and the unit stated (or implied) in the row label. The most common error: cells formatted as currency (displaying "$X,XXX") in rows that compute dimensionless "units of value" (not dollars). Units of value are dimensionless — a row labeled "Added units of value due to..." or "Counterfactual units of value..." should show plain numbers (e.g., "26,133"), not dollar amounts ("$26,133"). Similarly, rows labeled "% improved spend" or "Probability" should show percentage format, not plain decimals. To check: compare the FORMATTED_VALUE output (which reflects applied cell formatting) against the row label's stated unit. Flag any row where currency formatting appears on a units-of-value row, or where plain decimal formatting appears on a row whose label says it is a percentage or rate. Severity: Low/O. Note: this error is invisible in FORMULA mode and only detectable by comparing FORMATTED_VALUE output to row labels.

**Terminology**: Flag every instance of "x cash" or "GiveDirectly" in row labels, cell notes, or results rows. Should read "x benchmark" per updated benchmark policy (Nov 2025). This rule applies to all vetted workbooks including internal BOTECs and working documents — do not add qualifiers like "this may not apply to internal docs." The terminology policy applies to GiveWell workbooks generally.

**Vestigial pass-through rows**: Flag any row labeled "after adjustments," "units of value after X," or similar where the formula simply equals the row above it (e.g., `=B19`) with no adjustment actually applied. These rows exist to hold future adjustments but currently add no information and can mislead readers into thinking adjustments were made. Flag as Low and recommend either applying an adjustment or removing the row.

**Unnecessary function wrappers**: Scan all formula cells for function wrappers that add visual complexity without changing the result. Sample every 5th formula row if the sheet has more than 50 formula rows — a representative sample is sufficient for this Low/Legibility check. Make targeted FORMULA mode reads for sampled rows only. Common patterns: `=SUM(X)` where X is a single cell or a simple arithmetic expression that does not benefit from SUM; `=IFERROR(X, X)` where both arms of the IFERROR are identical. These patterns typically indicate a formula was copy-pasted from a multi-cell range and never updated, or that error-handling was added without reviewing the fallback. File as **Low/O** in Publication Readiness (Legibility): "[cell] uses `=SUM([single expression])` — SUM is unnecessary here and should be replaced with `=[expression]` directly." Group all instances in one finding. No CE impact, but unnecessary wrappers obscure intent and signal the formula was inherited without review.

**Leverage/funging sign and framing**: Read every leverage and funging row label as a first-time external reader. Flag any case where a negative adjustment is described with language that sounds positive (e.g., "crowding funding into the program"), or where the sign is not self-evident from the label alone. The clearer framing is "adjustment for the opportunity cost of crowding in other resources" rather than "leverage," and "adjustment for the possibility of crowding funding out" rather than "funging." If the label does not make the direction of the sign obvious, flag as a Low finding and recommend adding a parenthetical explanation — e.g., "(negative because crowded-in resources have an opportunity cost elsewhere)."

**Formula decomposition — packed cells**: When a single cell combines 3 or more distinct logical operations (e.g., data lookup + adjustment factor + unit conversion in one expression), flag as Low/O and recommend breaking into intermediate rows with descriptive labels. Example: `=VLOOKUP(A3,rates!B:D,2,0)*0.75*C8/1000` packs a lookup, two scaling factors, and a unit conversion into one cell — a reader cannot verify any step without reverse-engineering the whole formula. Preferred pattern: one row per logical step, each with a label. Note: this is a legibility recommendation, not an error — do not escalate above Low unless the packed formula also lacks a walkthrough note (in which case the complex formula check below also applies at Medium/H).

**Complex formula walkthrough notes**: When a VOI tab or other sheet contains formulas with multiple nested functions (AVERAGEIFS with multiple criteria, SUMPRODUCT, OFFSET, complex SUMIFS), verify there is a cell note or adjacent label explaining what the formula calculates in plain language. A formula like `=AVERAGEIFS('CEA1'!$8:$8,'CEA1'!$41:$41,">"&B$9,'CEA1'!$1:$1,"<>Nigeria")/1000000/2` with no note is a Low finding — a reader cannot verify the logic without opening the cell and tracing each component. Recommend a brief note: "Average cost-effectiveness across countries meeting the [Nx] bar, excluding Nigeria, converted to millions."

**Multi-tab and non-obvious algebra notes**: When a formula (a) draws inputs from 3 or more distinct cells across different rows or tabs, or (b) implements non-obvious mathematical operations — present-value discount chains, counterfactual-coverage derivations, weighted averages of estimates that are themselves weighted — flag as Low if no cell note walks through the computation in plain language. Exception: if the formula is in the direct CE output chain (within 2 hops of the CE ratio cell) and its missing note prevents a reviewer from verifying the CE calculation, escalate to Medium. File as Legibility (column D blank — Publication Readiness routing). This is distinct from the nested-function check above: the trigger here is comprehension difficulty, not formula complexity. A reader should be able to verify the formula's logic from the note without tracing every cell reference. Recommended note format: "Calculates [X] as [plain English description]. Inputs: [cell ref] (description), [cell ref] (description)." Example triggers: a VOI present-value formula drawing on wait time, discount rate, and scenario probability across multiple rows; a counterfactual coverage formula deriving a net coverage gap as a ratio of ratios.

**Internal naming consistency**: Compare the program or team name, acronym, and intervention description used throughout the spreadsheet against the grant description provided in program context (Step 0.5). Flag any instance where the spreadsheet uses a different acronym or name than the grant — e.g., "IPM team" when the grant refers to "PPM team," or "malaria prevention program" when the grant page says "SMC program." Flag as Low/O. This check is most important after a program has been renamed or restructured: stale names often persist in row labels and cell notes after the official name changes.

**Internal terminology consistency**: As part of the row-by-row label scan, flag when the same concept or population group is referred to by different terms in different rows of the same sheet. Examples: "PLHIV" in one row and "HIV patients" or "people living with HIV" in another; "beneficiaries" in one row and "participants" or "recipients" in another; "treatment" in one row and "intervention" in another for the same programmatic activity. This is distinct from internal naming consistency (which compares against an external grant document) — this check looks for inconsistency *within* the spreadsheet itself. Inconsistent terminology forces a reader to infer whether two labels refer to the same population or concept, increasing misread risk. File as Low/O listing all rows where inconsistent terms appear. Recommended fix: standardize to whichever term appears most frequently, or to the term used in the grant description if available. Exception: do not flag when a distinction is intentional and meaningful (e.g., "enrolled beneficiaries" vs. "reached beneficiaries" if these represent different populations in the model).

**Verbatim template language**: Flag any row title, note, or parameter description copied verbatim from the VOI/optionality template without customization to this specific program.

**Stale within-workbook tab references in notes**: When a cell note references another sheet or tab by name (e.g., "see the country-specific 'Methods' tabs," "see Supporting Analysis"), verify the referenced tab exists and covers the same year or period as the current data. If the referenced tabs cover only a prior year (e.g., 2019–2021 Methods tabs exist but the note is in a 2022 results tab), flag as Low/O: the note should be updated to clarify which year's tabs are referenced.

**Undocumented analytical scope restriction**: In aggregate results tabs where the analysis is restricted to a subset of the data (e.g., only 4-cycle areas in a dataset that includes both 4- and 5-cycle areas, only certain geographies in a multi-country model), verify that a cell note or nearby row label explains the scope restriction. If a reader could reasonably expect the full dataset to be included and the restriction is undocumented, flag as Low/O and recommend adding a brief scope note.

**Row description with no values**: When a row has a descriptive label in column A implying it contains a calculated value (e.g., "Average coverage across all cycles," "Weighted proportion") but all value cells in that row are empty, flag as Low/O. Ask the researcher to either populate the row or remove the description — an unpopulated description row can mislead a reader into thinking a calculation exists.

**Date serial numbers as text labels**: When scanning row labels and cell values in FORMATTED_VALUE mode, flag any cell whose displayed value is a 5-digit integer (e.g., 45791, 44927) appearing where a text label is expected — particularly in age-group label cells (e.g., a cell that should read "5-14"), period descriptions, or category names. This occurs when a date is accidentally entered into a text-labeled cell in Excel/Sheets and stored as a serial number. File as **Low/O** in Publication Readiness (Legibility): "Cell [ref] displays a numeric date serial ([value]) instead of the expected text label. Convert to the intended text (e.g., '5-14' for an age-group label)."

**Staff first names in source citations**: When scanning cell notes and source columns, flag any note that cites a GiveWell staff member by first name only as the source of a value — e.g., "per Jack's model," "Bea's analysis," "from Meghna's spreadsheet," "per CRA." These are not traceable publication-ready citations. File in Publication Readiness as Sourcing: "Source note at [cell/row] cites a staff member by first name only ('[note text]') rather than a traceable document or analysis. Replace with a link to or title of the underlying document, or expand to full name and document reference." Do not flag notes that use first names in non-citation context already covered by Category E (first-person language) or Category H (action items addressed to a person).

**Placeholder cell content**: Flag any non-header data cell containing only dashes (`---`, `—`, `-`), `n/a`, `N/A`, or similar structural filler characters. These indicate template rows that were never populated or intentional blanks that should be documented. Check every data column, not just column A or the Notes column. Recommended fix: populate the cell, replace with an explicit blank and a Notes entry explaining why it is empty, or delete the row if it is structural scaffolding that was not removed. Flag as Low/O.

**Range-without-rationale notes**: When a hardcoded value row's note describes a range of external data points (e.g., "payments ranging from $112 in Nigeria to $225 in Mozambique," "values of X across Y countries," "estimates between X and Y") without then stating why this specific value was chosen, flag as Low/O. A note that only describes what the data says — without explaining why this row's value was selected from within that range — is incomplete documentation. The note should include a sentence explaining the selection rationale: e.g., "We chose the lower bound as a conservative estimate," "We use the midpoint as a representative value," "We chose this value because it most closely matches our target context." Check every hardcoded row whose note contains the words "ranging," "range," "between X and Y," or lists multiple data points from different sources.

**First-time reader final scan**: After completing all named checks above, perform one final top-to-bottom pass through every fully vetted sheet asking a single question for each row: *"Would a first-time external reader — someone with general GiveWell knowledge but no specific context on this model — misunderstand what this row does or represents?"* This catches intuitive misreads that don't match any named pattern: wrong org name embedded in a label, a stale geographic scope, an adjustment described by its mechanism rather than its effect ("leverage" instead of "adjustment for crowding-in opportunity cost"), a section that makes sense to the author but requires internal context to parse. When a row causes you to pause — even briefly — flag it as Low/O. Do not skip rows because you already checked the label in an earlier pass; this scan asks a different question than label precision.

## Step 7b — Cross-Sheet Consistency

**CEA ↔ VOI**: The VOI sheet's "direct benefits CE" parameter should match (or be explicitly reconciled with) the CEA sheet's bottom-line CE output. If they differ without explanation, flag it.

**Shared parameters**: Any parameter appearing in multiple sheets (grant cost, moral weights, benchmark, discount rate) should have the same value in all sheets, or an explicit documented reason for any difference.

**Multi-location shared parameter duplication**: In models covering multiple geographies (columns), parameters identical across all locations — discount rate, moral weights, GiveDirectly benchmark, overhead %, intervention-level adjustments — should feed from a single dedicated shared-parameters row rather than being hardcoded independently per column. Independently hardcoded identical values create a maintenance hazard: updating one column without updating the others silently diverges the model. Scan all rows where every column holds the same hardcoded value; if no single source row exists that the columns reference, flag as Medium/H and recommend consolidating into one source row with per-column formulas referencing it. Exception: if identical values reflect genuinely independent estimates that happen to agree, document this with a note.

**Parallel program sheets**: When two or more sheets model the same intervention type with a stated ratio relationship between parameters, verify that ratio holds for every shared structural parameter (at-risk duration, active risk frequency, time horizons). Flag undocumented asymmetries — differences can be intentional but must be documented.

**Structural row-parity for parallel tabs**: When two tabs model the same intervention with parallel structure (e.g., Lenacapavir and Oral PrEP), verify not just that parameter values are consistent but that the computation steps themselves are arranged identically — same number of rows in each logical section, same row-to-row logic. Structural asymmetries (a section that has 3 rows in one tab but 4 in another, or steps in a different order) make the tabs harder to audit side-by-side and often indicate one tab was updated while the other wasn't. Flag row-count or logic-order divergences as Low and recommend standardizing the layout so the tabs can be read in parallel.

**Same-term cross-sheet meaning check**: When the same row label or variable name appears in multiple sheets, verify the concept it represents is consistent. A term that means one thing in one tab (e.g., "marginal WFP-attributable intake") and something different in another tab (e.g., "total fortification intake vs. baseline") must be flagged as Low/O — inconsistent term definitions force readers to infer meaning from context and create systematic misreading risk. Recommended fix: update the label in one or both tabs to be specific about what population or quantity each measures, and add a cell note explaining the relationship between the two measures.

**"High-level only" sheets**: When the user asked you to review a sheet at a high level only: (a) read top-line output values, (b) verify every cell referenced by the main vet sheet uses the expected value and range, (c) check that the sheet's structure matches what the main sheet assumes. Do not catalog every formula, but do flag anything in (a)–(c) that looks wrong.

## Writing Findings

Before writing any finding, confirm you can answer all three of these: (1) the exact cell reference(s) or row affected, (2) the specific label, note, or structural issue, and (3) the precise recommended fix. A finding that says "this section needs better labeling" without naming a cell is not complete.

**Before filing any Assumption or Inconsistency finding**: ask: "What would a researcher who trusts this value point to as their evidence?" Write it as a single sentence in your reasoning before deciding whether to file. Only after writing that sentence, test it against the available evidence. If the defense holds up even partially, downgrade severity. If it fails, file with confidence.

Their sole purpose is to let the reconciliation agent confirm this instance completed normally without a silent failure (auth timeout, context limit, API error).
