**Stakes — why this matters**: GiveWell allocates hundreds of millions of dollars in grants based on cost-effectiveness analyses like this one. A missed formula error, a stale parameter, or an uncaught copy-paste bug can cause CE estimates to be overstated by 2–10×, directing funding toward less effective interventions or away from more effective ones. Every finding you miss here could affect real funding decisions and, ultimately, lives. Exhaustive coverage is the baseline requirement — not a stretch goal. Exhaustion is not an excuse for stopping early. The Role calibration block below governs how to *classify* what you find — not how thoroughly to look for it. Thorough coverage and conservative severity are both required.

**Role calibration**: GiveWell does not treat cost-effectiveness estimates as literally true — deep uncertainty is inherent to all CEAs. Your role is to catch genuine errors and surface undocumented assumptions, not to second-guess defensible modeling choices. When a researcher's approach is reasonable but undocumented, prefer a clarifying question (Low) over a finding (Medium/High). Reserve Medium and High for factual errors, internal inconsistencies, or missing required elements.

**Coverage mandate — no shortcuts**: Every check in this file applies exhaustively to all rows, all columns, and all cell notes across every vetted sheet — no sampling, no section-restricted scope, no "key cells only." After completing each major check section, write a coverage declaration in the canonical pipe-delimited format from SKILL.md: `COVERAGE | consistency-check | [check name] | [rows/cells checked] | issues found: [N] | status: complete`. An agent that stopped early cannot produce these declarations accurately — that is the point. Do not proceed to the next section until you can write the declaration.

## Step 4b — Cross-Parameter Consistency

**Missing-row handling**: When scanning for expected rows (based on sheet structure or header labels), if a row expected to be present is absent from the FORMATTED_VALUE cache, note this as a coverage gap rather than silently skipping it. Write a coverage note in your reasoning: "Row [label] not found in cache — check may be incomplete." Do not file this as a finding; record it in the coverage declaration so the reconciliation agent can assess whether the absence is structural or a scan gap.

**Study-derived parameter row consistency**: When a row is labeled as deriving a specific parameter from a single study (e.g., "Implied reduction in NMR from pre/post study," "Effect size from [trial name]," "Observed [outcome] rate"), verify that all geography or scenario columns reference the same source derivation — not independently re-computed versions of the parameter for each column's own baseline. A common error: column B computes `=B20/B18` (deriving the study effect using column B's own baseline), and columns C–F also compute `=C20/C18`, `=D20/D18`, etc., each producing a different implied effect. If the study result is a single fixed number, columns C–F should all reference `=$B$[row]` (or the equivalent), not recompute the ratio using geography-specific denominators. Flag as **Medium/H** when: (a) the row label indicates a single-study result and (b) columns use independent derivations that produce different values without a note explaining why. Ask whether each column is intended to use a geography-specific version of the parameter or the single study result.

**Shared-denominator check for ratio formulas**: When a row contains ratio formulas (e.g., `=B5/B3`), check whether all parallel columns (C, D, E, …) in that same row use a consistent denominator reference. A shared denominator (e.g., a single study-derived baseline) should appear as an absolute reference (`$B$3`) in every column's formula; using relative references (each column divides by its own local baseline, producing `=C5/C3`, `=D5/D3`, etc.) silently produces different implied ratios for each geography when the intent is to hold the denominator fixed. Scan every ratio formula row: read each affected cell in FORMULA mode, confirm whether the denominator reference is absolute or relative, and check whether the values actually differ across columns as a result. Flag as **Low/Inconsistency** when: (a) denominator references are mixed (some columns use `$B$3`, others use a local relative reference) without a cell note explaining the mixed treatment, OR (b) all columns use relative references on a row whose label indicates a single fixed parameter (e.g., "Implied effect from [trial name]," "Study-derived rate"). Do not flag when relative references are uniform and the row label clearly indicates a geography-specific recalculation is intended.

Coverage declaration: `COVERAGE | consistency-check | shared-denominator check | [all ratio formula rows across all vetted sheets] | issues found: [N] | status: complete`

**Unit consistency check — same parameter, different unit representations**: Scan for cases where the same parameter appears in multiple sections of the workbook with different unit representations. Common failure: coverage rate stored as a proportion (0.87) in one section and as a percentage (87%) in another, with no unit conversion in the formulas that reference both. Scan for:
- The same parameter label appearing with values that differ by a factor of ~100 across sections (proportion vs. percentage confusion).
- Formulas that combine a proportion-format value with a percentage-format value without a ×100 or ÷100 conversion factor (e.g., `=B12 * C7` where B12 is 0.87 and C7 is 87).
- The same named parameter appearing in multiple tabs with inconsistent scale (e.g., a coverage rate is 0.82 in the Main CEA tab and 82 in a source data tab that is referenced by the Main CEA tab).

Flag as **Medium/H** if the unit inconsistency would affect the CE calculation. Flag as **Low/H** if the inconsistency is in a supporting section that does not directly drive the CE output.

Coverage declaration: `COVERAGE | consistency-check | unit consistency check | [all rows across all vetted sheets] | issues found: [N] | status: complete`

**Declared-intentional deviation verification — do this before other parameter checks**: For each item in the session context "Declared-intentional deviations" list, verify the deviation is actually present in the spreadsheet before skipping that cell in the parameter consistency check:

1. Identify the cell or row the deviation applies to.
2. Read that cell using `read_sheet_values` (UNFORMATTED_VALUE mode).
3. Confirm the actual stored value matches the declared deviation.

**If the declared value matches**: proceed to the magnitude check below before skipping the cell in the parameter consistency check.

**If the declared value does not match**: file as **High/D**. Explanation: "Declared deviation states [X] but cell [ref] contains [actual value]. The declared deviation cannot be verified against the spreadsheet." This is a finding regardless of which value is correct — an unimplemented or differently-implemented deviation must be resolved by the researcher.

**If no cell reference was given for a deviation** (e.g., "we use a different discount rate"): attempt to locate the cell by scanning for the relevant parameter row. If the cell cannot be located, file as Low/H — ask the researcher to confirm the exact cell so it can be excluded from parameter consistency findings.

**Magnitude check — applies to every confirmed deviation**: After confirming a declared deviation is present, check whether the declared value falls within GiveWell's documented defensible range for that parameter. Use `reference/key-parameters.md` and the Cross-Cutting CEA Parameters doc as the range reference. If the deviation falls outside the documented range, file as **Low/H** — do not escalate to Medium/High, but require researcher acknowledgment. Explanation template: "Declared deviation [X] falls outside GiveWell's documented range for this parameter ([Y]–[Z] per key-parameters.md). Researcher has confirmed this is intentional — please confirm the value is appropriate for this specific program context." Examples of out-of-range triggers: p(update) > 50%, discount rate < 0% or > 6%, probability of trial failure > 90%, other-funders benefit share > 50%, moral weight deviation > 20% from published value. If `key-parameters.md` does not define a range for the parameter in question, skip the magnitude check for that deviation.

If no declared-intentional deviations exist in session context, skip this step.

**TA PV-factor discounting consistency**: When the workbook is a TA BOTEC, verify that the PV (present value) discount factor used in the TA CEA matches the discount rate in the main model. Specifically: (a) locate any PV-factor rows in the TA tab (rows with labels containing "PV factor," "present value factor," "discount factor," or similar); (b) identify the discount rate those factors imply (by back-calculating from the factor or reading the discount-rate cell they reference); (c) compare against the discount rate used in the main CEA's benefit calculation. An inconsistency arises when the TA tab uses a different PV factor than what the main model applies to the same benefit stream type — a common error when a TA tab is copied from a different program with a different discount rate. Flag as **Medium/Inconsistency**: "TA tab uses a PV factor implying a [X]% discount rate while the main model uses [Y]% for the same benefit stream type. Inconsistent PV factors between TA and direct effects invalidate the CE comparison." If both use the same discount rate, note this as confirmed in the check log.

**TA model discount rate by benefit stream**: When the workbook is a Technical Assistance (TA) BOTEC, verify the discount rate applied is appropriate for each benefit stream type: (a) **death-averting streams** should use **1.4%** (GiveWell's temporal uncertainty component, not the full general discount rate); (b) **income or consumption streams** should use **4%** — a benefit stream is an "income or consumption stream" only when the model directly measures a change in earnings, wages, or consumption expenditure (a row labeled "increase in household income," "earnings gain," or "consumption impact" with a dollar amount or income multiplier); cognitive, educational, or developmental outcomes are death-averting streams (1.4%) even when the literature links them to future earnings, unless the model contains an explicit dollar-denominated income row; (c) mixed models (e.g., IFA with both morbidity and cognitive/income benefits) should apply 1.4% and 4% separately to their respective streams, not a single blended rate. Flag as **Medium/H** if: a death-averting TA model uses 4% (overstates discounting, understates CE); an income stream uses 1.4% (understates discounting, overstates CE); or a mixed-benefit model applies one uniform rate across all streams without a note documenting the choice. When the benefit stream type cannot be determined from row labels alone, file **Medium/H** — ask the researcher to confirm which rate applies — rather than silently applying one rate. To identify the applied rate, read the discount rate cell(s) in FORMULA mode and trace to the parameter row — do not rely on row labels alone, as the label may say "discount rate" without specifying which stream it applies to.

**TA discount rate — numerator/denominator symmetry**: Beyond verifying the correct rate type, check that discounting is applied *symmetrically* between the numerator (UoV or benefit calculation) and the denominator (population or coverage estimate). Common failure mode in TA BOTECs: the model discounts a multi-year population or coverage time series in a supporting tab (e.g., using a PV-adjusted average population as the denominator) while the UoV formula uses undiscounted adverse-outcome counts — or vice versa. If the denominator is discounted but the numerator is not, CE is overstated (denominator shrinks without a corresponding benefit reduction). To check: (a) read the formula for the population/denominator row in FORMULA mode and look for any multiplication by a discount factor or reference to a discounted time series; (b) read the UoV or benefit formula and check whether it applies a corresponding discount; (c) confirm the two treatments are consistent. Flag as **Medium/H Inconsistency** if one side is discounted and the other is not, with no cell note explaining the asymmetric treatment. Ask whether the discount treatment is intentional and symmetric across all components of the CE calculation.

## Writing Findings

**Severity guard**: Before filing a finding that classifies a specific hardcoded value as *wrong* (High/D), you must have done at least one of: (a) read the value in the Cross-Cutting CEA Parameters doc and confirmed the spreadsheet value contradicts it, (b) verified against `reference/key-parameters.md`, or (c) confirmed via arithmetic the value is internally inconsistent. Do not file a value-error High/D based solely on reasoning about what the value "should be." If the source is inaccessible or the correct value is uncertain, downgrade to Medium/H.

Before writing any finding, confirm you can answer all three of these: (1) the exact cell reference(s) affected, (2) the specific parameter that deviates and the authoritative value it should match, and (3) the precise fix required.

**Before filing an Assumption-type finding**: For each finding you are about to file as an Assumption (where the value may be intentional rather than wrong), ask: "What would a researcher who trusts this value point to as their evidence?" Write it as a single sentence in your reasoning before deciding whether to file (e.g., "Strongest defense: the VOI template default was updated after this model was built"). Only after writing that sentence, test it against the available evidence. If the defense holds up even partially, downgrade severity rather than filing Medium or High. If it fails, file with confidence. Do not skip this step for Assumption findings — the alternative is filing questions as errors. A finding that flags a deviation without naming the cell and the expected value is not complete.

Append findings using `modify_sheet_values` to your staging sheet. Start at row 2 and append sequentially. Your staging sheet name is provided in session context.
