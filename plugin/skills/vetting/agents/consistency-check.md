## Step 4b — Cross-Parameter Consistency

**Study-derived parameter row consistency**: When a row is labeled as deriving a specific parameter from a single study (e.g., "Implied reduction in NMR from pre/post study," "Effect size from [trial name]," "Observed [outcome] rate"), verify that all geography or scenario columns reference the same source derivation — not independently re-computed versions of the parameter for each column's own baseline. A common error: column B computes `=B20/B18` (deriving the study effect using column B's own baseline), and columns C–F also compute `=C20/C18`, `=D20/D18`, etc., each producing a different implied effect. If the study result is a single fixed number, columns C–F should all reference `=$B$[row]` (or the equivalent), not recompute the ratio using geography-specific denominators. Flag as **Medium/H** when: (a) the row label indicates a single-study result and (b) columns use independent derivations that produce different values without a note explaining why. Researcher judgment needed ✓ — ask whether each column is intended to use a geography-specific version of the parameter or the single study result.

**Unit consistency check — same parameter, different unit representations**: Scan for cases where the same parameter appears in multiple sections of the workbook with different unit representations. Common failure: coverage rate stored as a proportion (0.87) in one section and as a percentage (87%) in another, with no unit conversion in the formulas that reference both. Scan for:
- The same parameter label appearing with values that differ by a factor of ~100 across sections (proportion vs. percentage confusion).
- Formulas that combine a proportion-format value with a percentage-format value without a ×100 or ÷100 conversion factor (e.g., `=B12 * C7` where B12 is 0.87 and C7 is 87).
- The same named parameter appearing in multiple tabs with inconsistent scale (e.g., a coverage rate is 0.82 in the Main CEA tab and 82 in a source data tab that is referenced by the Main CEA tab).

Flag as **Medium/H** if the unit inconsistency would affect the CE calculation. Flag as **Low/H** with Researcher judgment needed ✓ if the inconsistency is in a supporting section that does not directly drive the CE output.

Coverage declaration: "Unit consistency check complete. Parameters with multi-section appearances: [list]. Inconsistencies found: [list or 'none']. No other issues of this type."

**Declared-intentional deviation verification — do this before other parameter checks**: For each item in the session context "Declared-intentional deviations" list, verify the deviation is actually present in the spreadsheet before skipping that cell in the parameter consistency check:

1. Identify the cell or row the deviation applies to.
2. Read that cell using `read_sheet_values` (UNFORMATTED_VALUE mode).
3. Confirm the actual stored value matches the declared deviation.

**If the declared value matches**: proceed to the magnitude check below before skipping the cell in the parameter consistency check.

**If the declared value does not match**: file as **High/D**, Researcher judgment needed ✓. Explanation: "Declared deviation states [X] but cell [ref] contains [actual value]. The declared deviation cannot be verified against the spreadsheet." This is a finding regardless of which value is correct — an unimplemented or differently-implemented deviation must be resolved by the researcher.

**If no cell reference was given for a deviation** (e.g., "we use a different discount rate"): attempt to locate the cell by scanning for the relevant parameter row. If the cell cannot be located, file as Low/H with Researcher judgment needed ✓ — ask the researcher to confirm the exact cell so it can be excluded from parameter consistency findings.

**Magnitude check — applies to every confirmed deviation**: After confirming a declared deviation is present, check whether the declared value falls within GiveWell's documented defensible range for that parameter. Use `reference/key-parameters.md` and the Cross-Cutting CEA Parameters doc as the range reference. If the deviation falls outside the documented range, file as **Low/H** with Researcher judgment needed ✓ — do not escalate to Medium/High, but require researcher acknowledgment. Explanation template: "Declared deviation [X] falls outside GiveWell's documented range for this parameter ([Y]–[Z] per key-parameters.md). Researcher has confirmed this is intentional — please confirm the value is appropriate for this specific program context." Examples of out-of-range triggers: p(update) > 50%, discount rate < 0% or > 6%, probability of trial failure > 90%, other-funders benefit share > 50%, moral weight deviation > 20% from published value. If `key-parameters.md` does not define a range for the parameter in question, skip the magnitude check for that deviation.

If no declared-intentional deviations exist in session context, skip this step.

**VOI structural completeness check — run before parameter values**: Load the Optionality/VoI BOTEC Template (`1wYsQZGsavXJQFSGF6Ea1k-p55C6dMbLPHhb0LKgNDZc`) and compare the workbook's row structure against the template's row list. **VOI parameter deviation check**: Load the Optionality/VoI BOTEC Template (`1wYsQZGsavXJQFSGF6Ea1k-p55C6dMbLPHhb0LKgNDZc`) and compare every key input parameter against its template default. Include the note text in the Explanation. Do not mark Won't Fix for undeclared VOI deviations regardless of note content. Parameters to check include: p(update) cap (≤50%), discount rate, trial failure probability (default 10%), wait time between study completion and policy update, other funders benefit share (default 20%), probability we're wrong (floor −10% for a very precise/strongly significant trial at t≈3; default −20% for a well-conducted RCT at t≈2; −30% at t≈1.7; −50% at t≈1.2; a value of 0% or positive is always wrong), how long we fund after study (default 10 years), CE of reallocated funding (default 1–2× above bar), bar scenario CE thresholds and their probability weights. **Optionality/VoI structural conformance check**: Using the Optionality/VoI Extensions Structure Reference (`1BYdNqrOu3jqVYHHQ5S2lNfq3vCXwid0CxhQf4SjSRqc`), check whether the workbook uses any complex optionality or VoI modeling methods — such as prior distributions, probability-weighted scenarios, or multi-stage updating. If it does, compare the sheet structure and formula logic against the reference: section layout, row organization, formula patterns for computing priors and posteriors, and aggregation logic. Flag any structural divergence as Medium or High depending on whether it would change the CE output. If the workbook does not use complex optionality/VoI methods, note this and skip the structural check.

**TA model discount rate by benefit stream**: When the workbook is a Technical Assistance (TA) BOTEC, verify the discount rate applied is appropriate for each benefit stream type: (a) **death-averting streams** should use **1.4%** (GiveWell's temporal uncertainty component, not the full general discount rate); (b) **income or consumption streams** should use **4%** — a benefit stream is an "income or consumption stream" only when the model directly measures a change in earnings, wages, or consumption expenditure (a row labeled "increase in household income," "earnings gain," or "consumption impact" with a dollar amount or income multiplier); cognitive, educational, or developmental outcomes are death-averting streams (1.4%) even when the literature links them to future earnings, unless the model contains an explicit dollar-denominated income row; (c) mixed models (e.g., IFA with both morbidity and cognitive/income benefits) should apply 1.4% and 4% separately to their respective streams, not a single blended rate. Flag as **Medium/H** if: a death-averting TA model uses 4% (overstates discounting, understates CE); an income stream uses 1.4% (understates discounting, overstates CE); or a mixed-benefit model applies one uniform rate across all streams without a note documenting the choice. When the benefit stream type cannot be determined from row labels alone, file **Medium/H** with Researcher judgment needed ✓ — ask the researcher to confirm which rate applies — rather than silently applying one rate. To identify the applied rate, read the discount rate cell(s) in FORMULA mode and trace to the parameter row — do not rely on row labels alone, as the label may say "discount rate" without specifying which stream it applies to.

**TA discount rate — numerator/denominator symmetry**: Beyond verifying the correct rate type, check that discounting is applied *symmetrically* between the numerator (UoV or benefit calculation) and the denominator (population or coverage estimate). Common failure mode in TA BOTECs: the model discounts a multi-year population or coverage time series in a supporting tab (e.g., using a PV-adjusted average population as the denominator) while the UoV formula uses undiscounted adverse-outcome counts — or vice versa. If the denominator is discounted but the numerator is not, CE is overstated (denominator shrinks without a corresponding benefit reduction). To check: (a) read the formula for the population/denominator row in FORMULA mode and look for any multiplication by a discount factor or reference to a discounted time series; (b) read the UoV or benefit formula and check whether it applies a corresponding discount; (c) confirm the two treatments are consistent. Flag as **Medium/H Inconsistency** if one side is discounted and the other is not, with no cell note explaining the asymmetric treatment. Researcher judgment needed ✓ — ask whether the discount treatment is intentional and symmetric across all components of the CE calculation.

## Writing Findings

**Before filing an Assumption-type finding**: For each finding you are about to file as an Assumption (where the value may be intentional rather than wrong), ask: "What would a researcher who trusts this value point to as their evidence?" Only after writing that sentence, test it against the available evidence. Do not skip this step for Assumption findings — the alternative is filing questions as errors. A finding that flags a deviation without naming the cell and the expected value is not complete.
