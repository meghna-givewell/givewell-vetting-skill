## Step 4 — Edge Case Pass

Scan specifically for edge cases that forward-reading passes commonly miss:

**Zero denominators**: Any formula dividing by a cell that could be zero — does the formula handle this? Flag cases where a zero denominator would silently produce a `#DIV/0!` or propagate as zero without alerting the researcher. Common locations: cost-per-outcome formulas, coverage fractions, and CE calculations. If an `IFERROR` or `IF(...=0,0,...)` wrapper already handles this, do not flag — verify the wrapper handles it correctly.

**Blank cell references**: Formulas referencing cells that are currently blank — does the formula produce a correct result if the cell stays blank, or does it silently treat blank as zero? A formula that multiplies by a blank cell treats blank as zero, which may be the intended behavior (geography not yet funded) or an error (a required input was forgotten). Flag as Medium/H if the blank-as-zero treatment would silently produce a wrong result. Flag as Low/H if blank-as-zero is plausibly intentional but undocumented.

**Negative values**: Subtraction formulas where the sign could flip — does a negative intermediate value propagate sensibly? Flag any case where a negative result would produce a nonsensical final output (e.g., negative CE, negative population count, negative coverage rate). Pay particular attention to adjustment formulas of the form `1 - rate` where `rate > 1` would go negative.

**Silenced errors**: `IFERROR` or `ISERROR` calls that return 0 or blank — verify the suppressed error is benign (e.g., a lookup over an intentionally sparse range), not masking a broken reference or a `#DIV/0!` from a real missing input. For each silenced error found, read the inner formula with FORMULA mode to determine what error it would show without the wrapper. If the suppressed error is `#REF!` or would indicate a broken cross-sheet link, flag as High/D. If it is `#DIV/0!` from a denominator that may legitimately be zero (inactive geography, not-yet-funded scenario), flag as Low/H.

**Aggregation boundary errors**: For `SUM` or `AVERAGE` ranges, verify the range doesn't inadvertently include or exclude a row when rows are inserted above or below the current bounds. Common failure: `=SUM(B3:B18)` where the section actually ends at row 17 or row 19. Check by reading the row labels at the stated range start and end, and confirming they correspond to the first and last entries in the section. A range that ends one row too early in a declining time series understates the sum; one that ends one row too late may include a totals row or a divider, double-counting.

**Sensitivity/threshold completeness** (mandatory when the workbook includes a sensitivity or threshold analysis tab): After completing the edge case scan, identify inputs whose ±50% perturbation would move the final CE estimate by ≥10%. To apply the test: for each hardcoded input in the CE formula chain, estimate whether halving or doubling the input would shift CE by ≥10% (a proportional estimate is sufficient — if the parameter enters CE as a direct multiplier, a 50% change produces approximately 50% change in CE unless substantially offset). For each such input, verify it appears as a varied parameter in the sensitivity or threshold analysis tab. An input that would materially move CE but is absent from the sensitivity tab is a structural gap. File as **Low/D** with Researcher judgment needed ✓: "[Input X] is not varied in the threshold/sensitivity analysis — a ±50% change would move CE by approximately [N]%. If this parameter is intentionally fixed, add a note explaining why; otherwise consider adding a breakeven column for it." Exception: do not flag an input whose cell note explicitly explains it is held fixed (e.g., "set by grant terms," "GiveWell standard not subject to sensitivity").

**Annuity-due vs. annuity-immediate**: For every `PV()` formula in all sheets, inspect the `type` argument. `type=0` is annuity-immediate (standard). `type=1` is annuity-due, which overstates PV by approximately `(1 + r)`. For every `PV()` formula where `type=1` and no cell note explains why beginning-of-period is appropriate, flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses PV() with type=1 (annuity-due), which applies payments at the start of each period and overstates present value by ~(1+r) relative to the standard annuity-immediate (type=0). Add a note if beginning-of-period timing is intentional; otherwise change type to 0." Note: VOI-specific PV() formulas are additionally covered by the formula-check-voi agent — do not duplicate findings for cells already covered there.

**SUMPRODUCT array constant audit**: For any formula using `SUMPRODUCT` with an inline array constant (e.g., `SUMPRODUCT({1,1,0,1}, B3:E3)`), flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses a hardcoded array constant `{…}` inside SUMPRODUCT — if columns are added or reordered, the array length must be updated manually. Verify the array length matches the data range and confirm that the zero-entries are intentional exclusions." This pattern is brittle: inserting a column shifts the data range automatically but leaves the constant array unchanged, causing silent dimension mismatches.

**INDIRECT and OFFSET formulas**: For any formula containing `INDIRECT()` or `OFFSET()`, flag as **Low/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses `INDIRECT/OFFSET`, which resolves the reference at runtime from a string or computed address rather than a fixed cell reference. These functions do not update when rows/columns are inserted or deleted, and they suppress standard dependency tracking. Confirm the formula is intentional and add a cell note explaining what reference is being constructed and why a direct reference cannot be used."

**Hidden rows and columns**: Scan the FORMULA mode read output for any row or column index that appears in adjacent formulas but is missing from the FORMATTED_VALUE output — this indicates a hidden row or column. For each hidden row or column, read its FORMULA values using `read_sheet_values` on that specific range. Flag as **Low/H** with Researcher judgment needed ✓ (**Structural Issue**): "Row/column [ref] is hidden and contains [formula/value]. Hidden rows can be inadvertently excluded from SUM/AVERAGE ranges when adjacent rows are visible, and their values can change without being noticed during review. Confirm the row is hidden intentionally and that all aggregation ranges explicitly include or exclude it as intended."

**INDEX/MATCH lookup key verification**: For any formula using `INDEX(..., MATCH(key, lookup_range, match_type))` to navigate a source tab, verify: (a) the `match_type` argument — when `match_type` is `1`, `-1`, or omitted (the default is `1`, approximate match ascending), the function silently returns the nearest value rather than an exact match, which has the same wrong-row failure mode as VLOOKUP approximate match; when `match_type` is `0`, proceed to key verification; (b) for exact-match (`match_type=0`) formulas, verify the MATCH key — whether a hardcoded string or a cell reference — exactly matches the intended identifier in the lookup range. A key of `"Kaduna"` in a range containing `"Kaduna - Cohort 17"` will resolve to the wrong row or return `#N/A`. For hardcoded string keys, read the lookup range and confirm the exact string appears. For cell-reference keys, confirm the referenced cell's value matches the corresponding column or row header in the source tab. Flag as **High/D** (**Formula Error**) if the key doesn't match the intended entry: "[cell] uses MATCH('[key]') but the source lookup range contains '[actual label]' — the lookup resolves to the wrong row/column." For approximate-match MATCH (match_type omitted, 1, or -1), flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses MATCH with match_type=[value/omitted] — approximate match. This requires the lookup range to be sorted ascending/descending and silently returns the nearest value on unsorted data. Change to match_type=0 for exact match, or confirm the lookup range is sorted and approximate match is intentional."

**VLOOKUP approximate-match audit**: For any formula using `VLOOKUP(key, range, col_index)`, inspect the fourth argument (`range_lookup`). When the argument is `TRUE`, `1`, or omitted (approximate match is the default in Google Sheets), the function requires the lookup column to be sorted ascending — on unsorted data it silently returns the wrong row with no error indicator. For every VLOOKUP in the workbook: if the fourth argument is missing or `TRUE`, flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses VLOOKUP with approximate match (`range_lookup` omitted or TRUE) — this returns the nearest-lower value, not an exact match. On unsorted lookup data this silently returns the wrong row. Change to FALSE for exact match, or confirm the lookup column is sorted and approximate match is intentional." If the fourth argument is `FALSE` or `0`, no finding.

**YEAR() applied to year-value check**: For any formula containing `YEAR()`, inspect the argument. If the argument resolves to or contains an integer in the range 1900–2100, Google Sheets interprets that integer as a serial date number rather than a calendar year — e.g., `YEAR(ROUND(J284,0))` where J284 = 2027 returns 1905, not 2027 (since day 2027 from Dec 30, 1899 falls in 1905). This produces a silently wrong result with no error indicator. Flag as **Low/H** (**Formula Error**): "[cell] applies YEAR() to a value of [X] — Google Sheets treats integers as serial dates, so YEAR([X]) returns [wrong year], not [X]. Remove the YEAR() wrapper and reference the cell directly." This pattern most often appears in trend or projection rows where a year value is stored in a cell and then wrapped in YEAR() to extract it — a no-op that corrupts the value.

**IFERROR / IFNA error-suppression audit** (CE-chain scope): For any formula of the form `=IFERROR(formula, fallback)` or `=IFNA(formula, fallback)` whose result feeds a CE-chain calculation — i.e., the cell's value is referenced downstream in the cost or benefit chain — flag as **Low/H** with Researcher judgment needed ✓ (**Formula Error**) when the fallback value is `0`, a blank, or any numeric literal: "[cell] uses IFERROR/IFNA with a numeric fallback of [value]. If the inner formula fails — e.g., due to a broken cross-sheet reference, a missing row, or a lookup miss — the cell silently returns [value] rather than surfacing an error. Confirm the inner formula cannot legitimately fail under current inputs, or replace the fallback with a string like 'ERROR — check source data' so failures are visible." Exception: do not flag `IFERROR(..., "")` or `IFERROR(..., "N/A")` used in display-only cells that are not part of the CE chain. Note: the **Silenced errors** check above in the Step 4 pass covers non-CE-chain IFERROR/ISERROR patterns — this check covers only CE-chain scope.
