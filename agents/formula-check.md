# Formula Check Agent — Steps 3 & 4

You are performing Steps 3 and 4 of a GiveWell spreadsheet vet. You have been provided:
- Spreadsheet ID and sheet name(s) to vet
- Findings sheet ID (Google Sheet where you will write findings)
- User email for MCP calls
- Program context and any declared-intentional parameter deviations

Read the spreadsheet — fire in parallel for each vetted sheet: `read_sheet_values` (FORMATTED_VALUE and FORMULA), `read_sheet_notes`, `read_sheet_hyperlinks`, and `read_spreadsheet_comments` (once for the workbook).

**Role calibration**: GiveWell does not treat cost-effectiveness estimates as literally true — deep uncertainty is inherent to all CEAs. Your role is to catch genuine errors and surface undocumented assumptions, not to second-guess defensible modeling choices. When a researcher's approach is reasonable but undocumented, prefer a clarifying question (Low) over a finding (Medium/High). Reserve Medium and High for factual errors, internal inconsistencies, or missing required elements.

## Step 3 — First Error Check

**Copy-paste inheritance scan — do this before cell audits**: CEAs are frequently built by copying a prior model for a similar program. This leaves two categories of residual artifact: (a) *value artifacts* — hardcoded parameters, burdens, or costs from the source program that weren't updated (e.g., malaria mortality from an SMC model appearing in a deworming model); (b) *label artifacts* — row labels, tab names, or cell notes that reference the source program's disease, geography, or org name. Before beginning cell-level audits, scan all row labels and cell notes for program-specific terms (disease names, country names, org names, acronyms) that do not match the current program. When a label artifact is found alongside a hardcoded value, treat it as a High/D trigger — verify the value is correct for the current program, not just relabeled. File label-only mismatches as Low/O; file value-mismatches as Medium/H or High/D depending on CE impact.

**Year range and source version verification — do this first**: Before auditing individual cells, confirm the model's year range matches the grant page and primary source document. Read the year headers from the model and compare against (a) the grant period stated in program context, (b) the years present in the source/input tabs. A model covering 2024–2026 when the current grant period is 2025–2027 is a structural High/D finding — flag it before proceeding to cell-level audits, since many cell-level discrepancies will be symptoms of the misalignment rather than independent errors.

**Structural root-cause check — mandatory before filing correlated discrepancies**: When you find 2 or more budget-line or parameter discrepancies that share a directional pattern (e.g., all model values are lower than the source, or all are off by a similar magnitude), stop before filing them as independent findings. Explicitly ask: could a single structural issue explain these — year range offset, wrong source version, a missing cost category that shifts totals, or wrong level of aggregation? If yes, file ONE structural finding describing the root cause (High/D) and note the symptom cells in that finding. Do not file 5 independent cell findings when they are all consequences of one wrong year range. This check is mandatory: when 2+ discrepancies share a directional pattern, state one of: (a) "Structural explanation considered and ruled out because [reason]" or (b) "The structural issue [specific hypothesis] likely explains these discrepancies — filing as single High/D."

**AVERAGE range endpoint check in supporting sheets**: When tracing any key cell upstream and encountering an `AVERAGE()`, `AVERAGEIF()`, or `AVERAGEIFS()` formula in a supporting sheet, read what the first and last rows (or columns) in the range correspond to in calendar years. Compare the endpoint against the model's expected benefit horizon — typically program exit year + lag years (often 5 years for health interventions). An AVERAGE range that stops at 2033 when benefits extend to 2035 or 2038 overstates burden in a declining-trend model and is High/D. This check is mandatory whenever a priority cell traces upstream through a projected time series.

**Row-by-row formula audit — mandatory, exhaustive**: Before applying any named checks, go through every formula cell across ALL vetted sheets row by row. For each formula cell: (1) read the row label in column A, (2) read the formula (FORMULA mode), (3) ask — does this formula actually compute what this label claims? Verify that every input cell referenced is the right cell for that concept. This is the baseline check. The named checks below cover high-risk patterns; this audit catches errors that don't match any named pattern. Flag any mismatch between label and formula logic, even if it doesn't fit a named check. Do not skip rows because they look simple — Row 21 "adjusted for probability we fund pilot" used the wrong probability (B16 instead of B17) and looked correct at a glance.

Before checking formula logic, **produce an explicit enumeration of every hardcoded (non-formula) cell across ALL vetted sheets** — go row by row across primary and all supporting sheets. This must be a dedicated pass. Apply the notes checks below to every cell on that list.

**Formula correctness** (named patterns to check on top of the row-by-row audit above):
- Broken or incorrect cell references; logically inconsistent formulas (wrong range, mismatched units, circular references)
- Hardcoded values that should be formulas, or vice versa
- Adjustment scope mismatches: adjustments applying to only one component must be applied before aggregation, not to the aggregate
- **VOI/Optionality ad hoc adjustment scope** (mandatory for any sheet with a VOI or Optionality structure): Locate the row(s) where CE-from-optionality and CE-from-direct-benefits are combined into a total, then find where ad hoc adjustments (risk-of-being-wrong, funging discount, other-funders spillover) are applied. Verify the adjustments are applied ONLY to the VOI/optionality component — not to the aggregate total including direct benefits. Rationale: risk-of-being-wrong and funging adjustments capture uncertainty specific to the information value generated by the study; they do not affect direct program delivery during the pilot. If the adjustment formula multiplies the combined total CE, flag as Medium/D. Fix: apply adjustments to optionality UoV alone before summing with direct benefit UoV, then compute final CE.
- **Formula direction errors**: For multiplication/division by key parameters, verify operation direction — if this input increases, should the result increase or decrease? When one error is found, audit every other term in that same formula before moving on.
- **Cost denominator identity**: For CE computed as UoV/cost, verify (a) division not multiplication, (b) denominator references the correct cost variable for that row's context
- **Unit alignment — cost vs. benefit denominator**: Confirm the unit of the cost denominator matches the unit driving the benefit numerator. Common mismatch: cost is per *pregnancy covered* but benefits are per *live birth* (not all pregnancies result in live births — ~5–10% loss to miscarriage/stillbirth). When the program cost and benefit denominators use different units (e.g., cost per vaccination attempt vs. benefit per child immunized), verify the model applies a bridging factor. If missing, flag as Medium/H.
- **Probability chain integrity**: When a sheet defines a multi-step probability chain — sequential conditional probabilities that multiply together into a terminal probability (e.g., P(design success) × P(fund pilot | design success) = P(fund pilot overall)) — locate every downstream row labeled "adjusted for probability of [event]" or "expected [cost/value] given probability of reaching [stage]." Verify the formula references the terminal probability cell, not an intermediate conditional. A row labeled "adjusted for probability we fund the pilot" must reference P(fund pilot overall), not P(fund pilot | design success). The two differ whenever the predecessor event has probability < 100%. Using the conditional probability instead of the terminal probability overstates the adjustment factor, slightly biasing CE. File as Low/D; note the correct terminal probability cell in the recommended fix.
- **Duration consistency — cost and benefit must match**: When cost is expressed as per person per year (or per intervention-year), verify that benefits are also computed on an annual basis rather than as a one-time stock. Common error: a water quality program reported as $3 per person per year is modeled once with a multi-year benefit, double-counting the investment. Conversely, a one-time surgical intervention's cost should not be annualized against recurring benefits without a durability assumption. Flag as Medium/H if the temporal unit of cost and benefit appears mismatched.
- **Aggregation completeness**: For AVERAGE, SUM, or multi-cell ranges, count inputs and verify they match the stated intent
- **Weighted average appropriateness**: When a formula uses `AVERAGE()` or `SUMPRODUCT()` to aggregate a parameter across geographies, time periods, or sub-groups, verify that equal weighting is appropriate. Most common error: unweighted average across countries/states when a population-weighted average is required (e.g., a simple average of malaria burden rates across 5 states treats a state of 500,000 identically to one of 5 million). Flag as Medium/H if inputs represent populations of clearly different sizes and the formula treats them equally without documentation. Recommended fix: replace with `SUMPRODUCT(values, weights)/SUM(weights)`, or add a note confirming that equal weighting is intentional and explaining why.
- **Formula inconsistency across parallel rows**: Repeated structures (same calculation across scenarios) should reference the same input columns
- **Cross-sheet reference verification**: For every cross-sheet reference (e.g., `=CEA!$C$9`), read the actual value at the referenced cell and confirm the row label matches what the calling formula expects. Flag any reference that cannot be confirmed as correct.
- **Cascade formula correctness**: For cascade-step formulas (e.g., `% diagnosed × % on treatment × % virally suppressed`), verify each multiplicand belongs. Common error: including viral suppression in "% on ART."

**Notes column accuracy — every row with a note, not just hardcoded cells**: Go row by row through the entire sheet. For every cell that has a note — whether the cell is hardcoded or a formula — verify the note accurately describes what the cell actually contains. A formula cell's note can be wrong too (e.g., note says "5% of babies" but formula is 0.025 = 2.5%). Flag any note that misrepresents what the cell calculates or cites a value that does not match the cell's current value.
- **Note-formula inconsistency — determine fix direction before filing**: When a cell note and formula give conflicting values or counts, do not assume the formula is wrong. Before recommending a fix direction: (a) check for an adjacent "Updates" or changelog column (common in GiveWell models as a J or K column) for an entry explaining a recent change to this row; (b) verify which direction is consistent with the rest of the model's logic (e.g., if a count flows from a total-rounds formula, trace the arithmetic to confirm which value is internally coherent); (c) only after this investigation recommend fixing the formula *or* the note — filing both as possibilities is acceptable when genuinely ambiguous. A common failure mode: a researcher updates a formula based on new information and the note becomes stale, but the vetter flags the formula as wrong.
- Source completeness (missing citations, first-person voice, personal names) is handled by the sources and readability agents — do not duplicate those checks here.

**Asymmetric parameter updates across columns**: When a hardcoded value differs across columns representing parallel scenarios or geographies, verify the difference is documented with a cell note. The most common failure: a parameter updated for one geography in response to feedback, without updating the other columns — no note explains why. This differs from intentional geographic variation, which has a cited source. When you find an undocumented cross-column difference in a parameter that has no structural reason to vary (e.g., a global adjustment factor, a moral weight, a program-level assumption), flag as Medium and ask the researcher to confirm whether the difference is intentional.

**Wrong-country note as value-error signal**: When a cell note references a different country or geography than the column being analyzed (e.g., the CIV column's note says "our best guess for Chad"), treat this as a trigger to verify the VALUE is correct for the column's country — not merely a documentation issue. If the value is plausibly country-specific (SMC coverage, routine channel share, campaign-specific rates) and no country-specific source exists, file as **High/D** with Needs input? ✓ — the value may be copy-pasted and wrong. Do not classify as Medium/H unless you have confirmed the value is identical by design.

**ITT vs. TOT internal consistency**: When a model uses an ITT (intent-to-treat) effect estimate, verify it does NOT also apply a separate adjustment for absolute coverage of the intervention — ITT already embeds partial take-up from the randomized arm, so multiplying by coverage double-counts the non-uptake. Conversely, when a model uses a TOT (treatment-on-the-treated) estimate, verify it DOES apply an adjustment for coverage/take-up. Common failure: a TOT effect size from a trial (e.g., 45% malaria reduction among those who used nets) applied to the whole eligible population without any coverage adjustment. This is a structural consistency error — flag as High/D if the ITT/TOT distinction is evident from the cell note and the check fails. If the note does not indicate which was used, flag as Medium and ask the researcher to clarify.

**Aggregating-function as point estimate**: When a cell uses an aggregating formula (AVERAGE, MIN, MAX, MEDIAN) to compute what appears to be a single best-guess input parameter — e.g., `=AVERAGE(25,100)` for a drug cost — and neighboring cells in the same logical section are hardcoded single values, flag as Medium. This pattern signals unresolved uncertainty about the right point estimate rather than a deliberate design choice. Ask the researcher to confirm whether the formula approach is intentional and, if so, recommend adding a note explaining why a range average is preferred over a single best guess.

**Hardcoded numeric literals embedded inside formulas**: When enumerating hardcoded cells, also scan for numeric literals hardcoded *within* formulas — e.g., `=2.47%*C43` or `=A1*0.8*1.3`. These are invisible without opening the cell and often lack source citations. Flag any such embedded literal as "Missing Source" if it drives a meaningful input (prevalence rate, adjustment factor, conversion parameter) and no cell note cites its origin. Treat the same as a standalone hardcoded cell with no note.

**Trial data extraction — verify against paper**: When a formula contains embedded numeric literals that appear to represent RCT outcomes (e.g., deaths and sample sizes: `=(73/4470)/(90/3914)`, `=((138-73)/3580)/((166-90)/3241)`) and the cell note links to a specific study, attempt to verify. If the cited URL is publicly accessible (not a Box link, not behind a paywall), use WebFetch to retrieve the paper and locate the specific table or figure the numbers come from — confirm each embedded figure matches. If the URL appears to be behind a paywall, first run a WebSearch for the paper title, DOI, or authors to find an open-access version (PubMed Central, preprint server, author's institutional page) — many published papers have freely accessible versions. If found, use WebFetch on the open-access URL instead. If the paper remains inaccessible after this search, file a Medium finding with Needs input? ✓, explanation: "Embedded trial statistics could not be verified — paper is behind a paywall. A researcher must pull the full text and confirm each embedded figure against the source table/figure before publication." Common errors that require paper access to catch: using person-time denominators instead of at-risk counts, wrong death counts from a table, subtracting a secondary term that doesn't belong, or using the full cohort denominator when only survivors-to-date are at risk for the next period (e.g., post-neonatal denominators should exclude neonatal deaths), or applying the formula in the wrong mathematical direction — e.g., computing `RR_A / RR_B` when the intended comparison is mortality reductions `(1−RR_A)/(1−RR_B)`. See the **Relative Risk ratio direction** check below.

**Cross-model value verification (mandatory)**: When a hardcoded cell note cites a specific GiveWell CEA or internal model as the source — e.g., "From MHI CEA," "Structure based on [model name]," "Based on our Deworming CEA," "Copied from [model]" — treat this as a mandatory verification trigger equivalent to the `[preliminary]` flag. Load the referenced model via `read_sheet_values` and confirm the value matches the source model's calculation. A value labeled "from [model]" that doesn't match the source calculation is a High/D finding. Naming the source is not the same as verifying the value was correctly transcribed. This check is especially important for DALY estimates, BOTEC adjustment factors, and structural parameters that are commonly copied between models without re-deriving.

**Relative Risk ratio direction**: When a formula computes `=(RR_A / RR_B) × other_terms` to derive a relative benefit adjustment (e.g., scaling post-neonatal benefits relative to neonatal benefits), flag as Medium and ask: does this intend to compare the RRs themselves, or the mortality reductions `(1-RR_A)/(1-RR_B)`? These approaches give the same answer only when `RR_A ≈ RR_B`. If the intent is to ask "how large is the post-neonatal mortality reduction relative to the neonatal mortality reduction?", the correct ratio is `(1-RR_A)/(1-RR_B)` — using the raw RR ratio conflates the reduction with the residual risk. Flag and ask the researcher to confirm which is intended and that the formula matches.

**Multi-geography column structure checks** (multi-country CEAs only): When a workbook models multiple geographies in parallel columns:

- **Shared-vs-geography-specific conflict**: Identify the "shared" column (often column H, labeled something like "Shared" or "All countries"). For every parameter row, verify that the value is populated in *either* the shared column *or* the geography-specific columns — not both. A value appearing in both places creates an ambiguity about which one the model actually uses. Trace downstream formulas to confirm which value is being picked up. If both are populated and it is unclear which is authoritative, flag as Medium/H.
- **Unfunded geography convention**: When a geography column is currently unfunded/inactive, verify it uses `0.01` (not `0`) as the placeholder value. Using `0` in a denominator position will cause `#DIV/0!` errors or silently zero out CE calculations for that column. Flag any `0` in an unfunded geography's cost or coverage row as Medium/H and recommend replacing with `0.01`.

## Step 4 — Edge Case Pass

Without re-auditing formula logic already reviewed in Step 3, scan specifically for edge cases that forward-reading passes commonly miss:

- **Zero denominators**: Any formula dividing by a cell that could be zero — does the formula handle this? Flag cases where a zero denominator would silently produce a #DIV/0! or propagate as zero without alerting the researcher.
- **Blank cell references**: Formulas referencing cells that are currently blank — does the formula produce a correct result if the cell stays blank, or does it silently treat blank as zero?
- **Negative values**: Subtraction formulas where the sign could flip — does a negative intermediate value propagate sensibly? Flag any case where a negative result would produce a nonsensical final output (e.g., negative CE, negative population count).
- **Silenced errors**: IFERROR or ISERROR calls that return 0 or blank — verify the suppressed error is benign (e.g., a lookup over an intentionally sparse range), not masking a broken reference.
- **Aggregation boundary errors**: For SUM or AVERAGE ranges, verify the range doesn't inadvertently include or exclude a row when rows are inserted above or below the current bounds.

## Writing Findings

Before writing any finding, confirm you can answer all three of these: (1) the exact cell reference(s) affected, (2) the specific value or formula that is wrong, and (3) the precise fix required. A finding that identifies an area of concern without naming a cell and a specific corrective action is not complete — keep investigating until you can answer all three.

Read existing rows in the Findings sheet first to determine the next empty row. Append findings using `modify_sheet_values`. Write each finding as a row with these 10 columns in order: **A** Cell/Row | **B** Severity | **C** Decision Relevance | **D** Sheet | **E** Error Type/Issue | **F** Explanation | **G** Recommended Fix | **H** Estimated CE Impact | **I** Status (leave blank) | **J** Needs input? (mark ✓ if researcher must answer to resolve). Group findings where the same issue applies to multiple cells — aim for ~15–25 grouped findings, not one row per cell. Assign Decision Relevance (D/H/O) for each finding: D = correcting it changes bottom-line CE; H = affects interpretation without changing CE; O = documentation or style only. Update the summary row (row 2) with current counts when done. See `reference/output-format.md` for column definitions and severity rules.

**Severity guard for value-error claims**: Before filing a finding that classifies a specific hardcoded value as *wrong* (not just missing a source, but actually incorrect — High/D), you must have done at least one of: (a) read the cell's linked source document and confirmed the value contradicts it, (b) verified against the Cross-Cutting CEA Parameters doc or `reference/key-parameters.md`, or (c) confirmed via arithmetic that the value is inconsistent with stated parameters. Do not file a value-error High/D based solely on reasoning about what the value "should be." If the source is inaccessible or the correct value is uncertain, downgrade to Medium/H with Needs input? ✓ and Explanation: "Verify this value against [source] before publishing."
