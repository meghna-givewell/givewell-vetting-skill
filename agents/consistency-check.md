## Step 4b — Cross-Parameter Consistency

**Cross-vet parameter consistency** (load Cross-Cutting CEA Parameters doc now):
Load the Cross-Cutting CEA Parameters doc (`1ru1SNtgj0D9-vLAHEdTM27GEq_P17ySzG-aTxKD6Fzg`) via `read_sheet_values` and **enumerate every parameter row in the doc**. Do not limit this check to a named list of parameters — check every row the doc contains, including SMC deaths-averted rates by geography, counterfactual spending rates, and any rows not listed in `reference/key-parameters.md`. For each row in the doc, locate the corresponding cell in the spreadsheet being vetted and compare values. The search for matching cells must be value-based — scan all hardcoded cells across the workbook for values that correspond to each doc row's parameter and geography, not only cells whose formulas explicitly reference the doc. A deviation of >5% from the cross-cutting doc value for the same geography is High/D regardless of whether the spreadsheet cell has a note citing the doc. A check limited to named parameters will miss rows that were updated in the doc after the model was built. Also check `reference/key-parameters.md` for the benchmark, discount rate, moral weights, and long-term income ratio. For declared-intentional deviations: file as `Intentional` / Severity `Low` with researcher's stated reason, Recommended Fix "No action required". For undeclared mismatches: file as `Outdated Parameter` / **High/D** — a geography-specific value that differs from the cross-cutting params doc for that same geography is a value error, not a documentation gap. Do not downgrade to Missing Source or Medium simply because a note exists; the severity is determined by whether the value is correct, not whether it is cited.

**Benchmark and final CE rows — always check, regardless of declared scope**: Even in targeted or heads-up vets where specific rows are declared in scope, always locate and check: (a) the GW benchmark row (typically labeled "Units of value per $, GiveWell benchmark" or similar) and (b) the final CE row (typically labeled "Final cost effectiveness (x cash/benchmark)"). These are the highest-stakes cells. Compare the benchmark value against `reference/key-parameters.md`. A stale benchmark is a High finding if it materially changes the CE multiple. When a researcher has declared a restricted row scope for this vet, note explicitly in your output that benchmark and final CE rows are being checked regardless — these rows are never excludable by scope declaration.

**VOI structural completeness check — run before parameter values**: Load the Optionality/VoI BOTEC Template (`1wYsQZGsavXJQFSGF6Ea1k-p55C6dMbLPHhb0LKgNDZc`) and compare the workbook's row structure against the template's row list. This is a presence check — verify that required structural rows exist in the model at all, not just that their values are correct. Flag any template row that is absent from the model as Low/H with Needs input? ✓. Key rows to check for presence: (a) **time-to-results** — months from grant start until results are available; if absent, flag: the model cannot assess whether the funding timeline is appropriate; (b) **wait time between results and policy update** — lag between study completion and funding decision; if collapsed to zero or absent, flag; (c) **probability of policy update given results** (p(update)) — must be present for each scenario and capped ≤50%; (d) **probability chain for design → pilot** — P(design success) and P(fund pilot conditional on success) should appear as separate rows, not collapsed. If the template row exists but the workbook collapses steps or omits them, flag and ask the researcher to confirm whether the omission is intentional.

**VOI parameter deviation check**: Load the Optionality/VoI BOTEC Template (`1wYsQZGsavXJQFSGF6Ea1k-p55C6dMbLPHhb0LKgNDZc`) and compare every key input parameter against its template default. Flag any deviation not listed as a declared-intentional deviation. Parameters to check include: p(update) cap (≤50%), discount rate, trial failure probability (default 10%), wait time between study completion and policy update, other funders benefit share (default 20%), probability we're wrong (default −20% for a well-conducted RCT; for lower precision studies: −30% at t≈1.7, −50% at t≈1.2), how long we fund after study (default 10 years), CE of reallocated funding (default 1–2× above bar), bar scenario CE thresholds and their probability weights. These parameters are easily inadvertently changed when a VOI tab is copied from a prior model — a deviation that appears plausible on its face may simply be a carryover from a different program context.

**Optionality/VoI structural conformance check**: Using the Optionality/VoI Extensions Structure Reference (`1BYdNqrOu3jqVYHHQ5S2lNfq3vCXwid0CxhQf4SjSRqc`), check whether the workbook uses any complex optionality or VoI modeling methods — such as prior distributions, probability-weighted scenarios, or multi-stage updating. If it does, compare the sheet structure and formula logic against the reference: section layout, row organization, formula patterns for computing priors and posteriors, and aggregation logic. Flag any structural divergence as Medium or High depending on whether it would change the CE output. If the workbook does not use complex optionality/VoI methods, note this and skip the structural check.

## Writing Findings

Before writing any finding, confirm you can answer all three of these: (1) the exact cell reference(s) affected, (2) the specific parameter that deviates and the authoritative value it should match, and (3) the precise fix required. A finding that flags a deviation without naming the cell and the expected value is not complete.

Read existing rows in the Findings sheet first to determine the next empty row. Append findings using `modify_sheet_values`. Write each finding as a row with these 10 columns in order: **A** Cell/Row | **B** Severity | **C** Decision Relevance | **D** Sheet | **E** Error Type/Issue | **F** Explanation | **G** Recommended Fix | **H** Estimated CE Impact | **I** Status (leave blank) | **J** Needs input? (mark ✓ if researcher must answer to resolve). Assign Decision Relevance (D/H/O) for each finding: D = correcting it changes bottom-line CE; H = affects interpretation without changing CE; O = documentation or style only. Update the summary row (row 2) with current counts when done. See `reference/output-format.md` for column definitions and severity rules.

**Severity guard**: Before filing a finding that classifies a specific hardcoded value as *wrong* (High/D), you must have done at least one of: (a) read the value in the Cross-Cutting CEA Parameters doc and confirmed the spreadsheet value contradicts it, (b) verified against `reference/key-parameters.md`, or (c) confirmed via arithmetic the value is internally inconsistent. Do not file a value-error High/D based solely on reasoning about what the value "should be." If the source is inaccessible or the correct value is uncertain, downgrade to Medium/H with Needs input? ✓.
