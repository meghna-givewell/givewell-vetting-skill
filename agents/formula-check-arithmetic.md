- Spreadsheet ID and sheet name(s) to vet
- Findings sheet ID
- User email for MCP calls
- Program context and any declared-intentional parameter deviations
- Sheet row scope: audit only rows `{scope_start}` to `{scope_end}`

A separate **formula-check-data** agent handles external data verification (GBD vizhub links, trial paper data, cross-model values, downstream re-computation). A separate **formula-check-edge-cases** agent handles the Step 4 edge case scan. Do not re-run those checks here. Source data tab plausibility is handled by dedicated source-data-check agents. **VOI-specific formula checks** (probability row column-reference consistency, cross-actor symmetry assumptions, VOI_Priors cross-formula scope, ad hoc adjustment scope) are handled by a dedicated **formula-check-voi** agent running in parallel — do not run those checks here.

Read the spreadsheet in batches — for each vetted sheet, call `read_sheet_values` using ranges `A1:ZZ50`, `A51:ZZ100`, `A101:ZZ150` (continuing in 50-row increments until a batch returns no non-empty rows), once in FORMATTED_VALUE mode and once in FORMULA mode. Fire `read_sheet_notes`, `read_sheet_hyperlinks`, and `read_spreadsheet_comments` (once for the workbook) in parallel with the first batch. **The MCP tool returns at most 50 rows per call regardless of range size — batches larger than 50 rows will silently drop data.**

**Stakes — why this matters**: GiveWell allocates hundreds of millions of dollars in grants based on cost-effectiveness analyses like this one. A missed formula error, a stale parameter, or an uncaught copy-paste bug can cause CE estimates to be overstated by 2–10×. Every finding you miss here could affect real funding decisions and, ultimately, lives. Exhaustive coverage is the baseline requirement — not a stretch goal.

**Role calibration**: GiveWell does not treat cost-effectiveness estimates as literally true — deep uncertainty is inherent to all CEAs. Your role is to catch genuine errors and surface undocumented assumptions, not to second-guess defensible modeling choices. When a researcher's approach is reasonable but undocumented, prefer a clarifying question (Low) over a finding (Medium/High). Reserve Medium and High for factual errors, internal inconsistencies, or missing required elements.

**Coverage mandate — no shortcuts**: Every check in this file applies exhaustively to all rows, all columns, and all cell notes across every vetted sheet within your assigned row scope. Read every row. Check every column. Read every cell note. Do not stop early when you have found several examples of an issue type. After completing each major check section, write two coverage declarations before moving on: (1) "Checked [rows X–Y / all N columns]. Found issues at: [list]. No other issues of this type." (2) "Read notes for rows X–Y: [N] notes found, issues at [list or 'none']." Do not proceed to the next section until you can write both. When a formula error is found in any row, continue checking the next 20 rows before moving on. For any IF, IFS, or SWITCH formula, verify each branch independently.

**Check priority**: Complete checks in this order to ensure the highest-value work is done first. Do not skip or sample any check — proceed sequentially and exhaustively through all four levels:
1. Copy-paste inheritance scan + year range verification + declared bug fix tracing + structural root-cause check
2. Cross-sheet reference inventory
3. Row-by-row formula audit (the core check — must complete before any named pattern checks below)
4. Named formula patterns (lines beginning "**Formula correctness**" through "**Parallel-column row-reference consistency**")

VOI-specific formula patterns (VOI probability row column-reference, cross-actor symmetry, VOI_Priors cross-formula, ad hoc adjustment scope) are handled by a dedicated `formula-check-voi` agent running in parallel — do not re-run them here.

## Step 3 — Formula and Arithmetic Check

**Copy-paste inheritance scan — do this before cell audits**: CEAs are frequently built by copying a prior model. This leaves two categories of residual artifact: (a) *value artifacts* — hardcoded parameters from the source program that weren't updated; (b) *label artifacts* — row labels, tab names, or cell notes referencing the source program's disease, geography, or org name. Before beginning cell-level audits, scan all row labels and cell notes for program-specific terms that do not match the current program. When a label artifact is found alongside a hardcoded value, treat it as a High/D trigger — verify the value is correct for the current program. File label-only mismatches as Low/O; file value-mismatches as Medium/H or High/D depending on CE impact.

**Year range and source version verification — do this first**: Before auditing individual cells, confirm the model's year range matches the grant page and primary source document. A model covering 2024–2026 when the current grant period is 2025–2027 is a structural High/D finding — flag it before proceeding to cell-level audits, since many discrepancies will be symptoms of the misalignment.

**Declared bug fix tracing — mandatory**: When the user's declared-intentional changes include bug fixes, treat each fix as a separate High-priority line item requiring a specific cell reference before this step closes.

1. Read `read_spreadsheet_comments` on the input spreadsheet first. Before proceeding, list every comment containing correction language ("corrected," "changed," "fixed," "was previously," "→") and map each to one declared fix by number. If correction comments exceed declared fixes, treat the excess as additional undeclared bug fixes. Proceeding without this enumeration is not permitted.
2. For each declared fix, read likely sheets in FORMULA mode, scanning for values matching the described correction.
3. After locating the cell: verify the corrected value is present, the formula logic is correct, and a cell note documents the change.
4. If the cell cannot be identified after reading all plausible sheets, file **High/H** with Researcher judgment needed ✓. Use the error type that best matches the declared fix description: **Formula Error** if a formula was changed, **Parameter Issue** if a value was updated, **Structural Issue** if the model structure changed.

**Structural root-cause check — mandatory before filing correlated discrepancies**: When you find 2 or more budget-line or parameter discrepancies sharing a directional pattern, stop before filing them as independent findings. Ask: could a single structural issue explain these — year range offset, wrong source version, a missing cost category, wrong level of aggregation? If yes, file ONE structural finding describing the root cause (High/D). When 2+ discrepancies share a directional pattern, state one of: (a) "Structural explanation considered and ruled out because [reason]" or (b) "The structural issue [specific hypothesis] likely explains these discrepancies — filing as single High/D."

**AVERAGE range endpoint check in supporting sheets**: When tracing any key cell upstream through an `AVERAGE()`, `AVERAGEIF()`, or `AVERAGEIFS()` formula in a supporting sheet, read what the first and last rows in the range correspond to in calendar years. Compare against the model's expected benefit horizon. An AVERAGE range that stops at 2033 when benefits extend to 2035 is High/D. This check is mandatory whenever a priority cell traces upstream through a projected time series.

**Cross-sheet reference inventory — do this before the row-by-row audit**: Before auditing individual formulas, produce a complete inventory of every cross-sheet reference in the workbook. For each unique referenced tab name, confirm:

1. **Tab exists**: the referenced sheet name appears in the workbook's tab list. A formula referencing a deleted or renamed tab resolves to `#REF!` — flag as High/D.
2. **Tab is current**: if the referenced tab name contains year or version information (e.g., "GBD 2021"), flag as Low/H if the year appears older than the current grant period.
3. **Concept match for high-stakes references**: For every cross-sheet reference in a formula whose row label describes a key parameter, read the row label of the referenced cell and confirm the concept matches. A formula labeled "GBD mortality rate under-5" pointing to a row labeled "all-ages mortality" is High/D. Pay particular attention to bare single-cell references of the form `=OtherTab!CellRef` — these are the highest-risk wrong-adjacent-row pattern, where the formula references row N when row N±1 was intended (e.g., `=Inputs!K676` referencing an exchange rate row when `=Inputs!K675`, the completion bonus row, was intended). When the referenced cell's row label is categorically different from the referencing cell's concept — different parameter type, different dose, different metric — flag as **High/D** (**Formula Error**) even when both rows are adjacent in the source tab. Do not require the rows to be far apart; adjacent-row wrong-reference errors are common and have no error indicator.

Write the inventory summary in your reasoning before proceeding: "Cross-sheet reference inventory: [N] unique referenced tabs. [issues or 'No further issues']." Do not proceed to the row-by-row audit until this declaration is written.

**Row-by-row formula audit — mandatory, exhaustive**: Read every formula cell across ALL vetted sheets in FORMULA mode within your row scope. For each formula cell:

1. Read the row label in column A.
2. Read the full formula.
3. **For each cell reference in the formula**: read the row label of the referenced cell and ask — does this label describe an appropriate input to this formula? This is a semantic check. You are verifying that the concept described by the referenced cell's label is the right concept for this formula. A formula labeled "dispenser treatment effect" that references a cell labeled "non-dispenser reporters" is a conceptual mismatch even if the arithmetic is valid.
4. Ask: does the formula as a whole compute what the row label claims?

Flag any mismatch at step 3 or 4, even if it doesn't fit a named check below. Do not skip rows because they look simple.

**Severity calibration for reference mismatches**: When step 3 identifies a cell reference pointing to a row whose label clearly doesn't match the referencing cell's concept — e.g., a "Rota dose 3" row referencing a "Measles dose 1" cell, or a "Completion bonus" row referencing an "Exchange rate" cell — call this a **clear formula error**, not a researcher judgment question. The structural label mismatch is itself the evidence. Reserve "Researcher judgment needed ✓" only when the mismatch could plausibly be intentional (e.g., using one dose as an acknowledged proxy for another with a documented reason). When the referenced row label is categorically different from the referencing row's concept, file at **High/D** (**Formula Error**) with the specific correction in column G.

**Absolute-reference note check — mandatory before filing any locked-column finding**: When any formula check identifies an absolute column reference (`$B$xxx`, `$C$xxx`, etc.) used uniformly across all geography columns — i.e., every column in the row references the same source cell — do not file a copy-paste finding until you have read the cell note on that row (check both the formula cell's note and the column A note). If the note explains the uniform reference is intentional (e.g., "we use [country] as the reference baseline because the study was conducted there," or "this parameter applies the same calibration across all geographies"), classify as intentional and do not file. When no note exists and the uniform reference is suspicious, file as Low/H with Researcher judgment needed ✓, not as High. Reserve High only when you can confirm via the note or external source that a country-specific value clearly differs from the locked reference.

Before checking formula logic, **produce an explicit enumeration of every hardcoded (non-formula) cell across ALL vetted sheets** within your scope — go row by row. Apply the notes checks below to every cell on that list.

**Formula correctness** (named patterns to check on top of the row-by-row audit above):
- **Circular reference detection**: For every batch of cells read in FORMULA mode, check for `#REF!` error values in formula text (e.g., a formula that contains `#REF!` due to a deleted cell reference). Also scan every batch read in FORMATTED_VALUE mode for cells whose displayed value is `#REF!` — this catches the more common case where a formula resolves to a `#REF!` error at runtime. Additionally, after the full read is complete, scan the returned values for any cell showing `0` or blank in a row where the formula clearly should produce a non-zero result — this is a common symptom of a Google Sheets circular dependency being silently resolved to zero. When a circular reference is suspected, verify by reading the dependency chain: read the formula of the suspected cell, then read the formula of each cell it references, checking whether any referenced cell transitively references back to the original cell. When a circular reference is confirmed, file as **High/H** with Researcher judgment needed ✓ (**Formula Error**): "[Cell ref] is part of a circular dependency — it references [cell X], which directly or transitively references it back. Google Sheets resolves circular references to zero or to the last cached value, making the CE estimate indeterminate. The circular link must be broken before this model can be trusted." Recommended Fix: identify which link in the dependency chain can be replaced with a hardcoded constant or a prior-step computed value that does not reference downstream cells; name that specific link in the fix.
- **Adjustment scope mismatches**: adjustments applying to only one component must be applied before aggregation, not to the aggregate. Flag as **High/D** (**Adjustment Issue**).
- **Formula direction errors**: For multiplication/division by key parameters, verify operation direction. When one error is found, audit every other term in that same formula before moving on. Flag as **High/D** (**Formula Error**) if the direction is definitively wrong; **Medium/H** with Researcher judgment needed ✓ (**Formula Error**) if intent is unclear.
- **Formula addend conceptual consistency**: For any formula that sums or combines multiple terms, read the row label of every referenced cell and verify each term is conceptually appropriate. A term measuring something categorically different from the other terms is a formula logic error. File as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**) when a formula mixes conceptually distinct quantities without a cell note explaining why.
- **"Implied" and "derived" row structural redundancy**: For any row labeled "implied," "sensitivity check," "derived," or "check," verify whether the formula independently re-computes a quantity already captured canonically by a single upstream input cell. If columns C–F each re-derive what column B already holds from a primary source (e.g., a paper estimate), flag as **Low/H** with Researcher judgment needed ✓ (**Structural Issue**): "Columns [refs] independently re-compute the quantity in [B ref]. If the paper estimate in column B is the authoritative value for all geographies, these columns should reference =$B$[row] directly. If the re-derivations are geography-specific, document why each column uses its own computation." This prevents formula fragility where the same underlying quantity is computed differently across columns without documentation.
- **Cost denominator identity**: For CE computed as UoV/cost, verify (a) division not multiplication, (b) denominator references the correct cost variable for that row's context. Flag as **High/D** (**Formula Error**).
- **Missing-multiplier disambiguation**: Before filing a finding that a formula is missing a factor (e.g., "cost × cohort is missing the patients-per-caregiver multiplier"), first verify whether the arithmetic already closes without it. Read the output row's displayed value and check: does `per_unit_cost × cohort_size` equal that value? If yes, the formula is computing correctly under some unit convention — and the row label ("per patient") may be stale rather than the formula being wrong. Check the cell note of the per-unit cost row: if it describes the cost as per-caregiver, per-trained-person, or per-cohort-member, file a label error (Readability, Publication Readiness) rather than a formula error. Do not propose "add multiplier B6" when the arithmetic already closes correctly without it.
- **Unit alignment — cost vs. benefit denominator**: Confirm the unit of the cost denominator matches the unit driving the benefit numerator. Common mismatch: cost is per *pregnancy covered* but benefits are per *live birth*. When the program cost and benefit denominators use different units, verify the model applies a bridging factor. If missing, flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**).
- **Probability chain integrity**: When a sheet defines a multi-step probability chain, verify downstream rows labeled "adjusted for probability of [event]" reference the terminal probability cell, not an intermediate conditional. Flag as **Low/H** with Researcher judgment needed ✓ (**Formula Error**); note the correct terminal probability cell in the recommended fix.
- **Duration consistency — cost and benefit must match**: When cost is per person per year, verify benefits are also computed on an annual basis rather than as a one-time stock. Flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**) if the temporal unit of cost and benefit appears mismatched.
- **Unexplained hardcoded scalar multipliers**: For any formula containing a numeric multiplier (e.g., `*2`, `*0.5`, `*0.1`, `*125`, `/2`) that is not a universally known constant (days per week, months per year, percentage conversion) and cannot be derived from the row label, cell note, or a standard structural constant, flag as **Low/H** with Researcher judgment needed ✓ (**Assumption Issue**) asking for documentation. Do not assume the multiplier is obvious — e.g., `*2` in a morbidity formula may double-count or may reflect a documented bilateral-effect conversion; a note is required either way. File at Medium/H if the multiplier drives more than 5% of the CE estimate.
- **Cascade formula correctness**: For any formula computing a multi-step product — a care cascade (e.g., `% diagnosed × % on treatment × % virally suppressed`), a benefit chain (`coverage × efficacy × discount factor`), or a probability chain (`P(seeking care) × P(correct diagnosis) × P(completing treatment)`) — verify that each multiplicand belongs at that step and does not conflate adjacent steps. Common error in HIV models: including viral suppression as part of the "% on ART" multiplier — these are distinct concepts (a person is on ART regardless of whether viral suppression has been achieved), so multiplying them together understates ART coverage and corrupts all downstream calculations. Ask: what does the row label say the output represents, and does each term in the formula contribute to exactly that concept and nothing more? Flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**) if a multiplicand appears to conflate adjacent steps.

**Redundant SUM wrapper**: Flag any formula of the form `=SUM(single_cell_reference)` or `=SUM(single_named_range)` where SUM is applied to a single non-range argument — file as **Low/O** in Publication Readiness (Readability). This pattern typically indicates a copy-paste from a multi-cell range that was never updated. No CE impact, but it signals the formula was inherited without review.

**Aggregation completeness**: For AVERAGE, SUM, or multi-cell ranges, count inputs and verify they match the stated intent.
- **Weighted average appropriateness**: When a formula uses `AVERAGE()` or `SUMPRODUCT()` to aggregate across geographies, verify equal weighting is appropriate. Flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**) if inputs represent populations of clearly different sizes and the formula treats them equally without documentation.
- **Formula inconsistency across parallel rows**: For repeated formula structures across parallel rows — the same calculation applied to different scenarios, cohorts, or disease components in separate row blocks — verify that all instances reference the same input columns and use the same formula structure. Flag any asymmetry where one row includes terms or column references that parallel rows omit (e.g., Scenario 1 formula references only column B, but Scenario 2 references both B and C). This is the row-dimension analog of cross-column structural consistency: a structurally unique row in an otherwise uniform section is typically a copy-paste error. Flag as **Low/H** with Researcher judgment needed ✓ (**Formula Error**): "[Row ref] in [section] uses [formula] while parallel rows [refs] use [formula pattern] — confirm whether the extra term is intentional."
- **Cross-column structural consistency within a single row**: For any row where all columns should share the same formula structure, scan for any column that deviates. A single outlier column in a horizontally consistent row is typically a copy-paste error. Flag as **Low/H** with Researcher judgment needed ✓ (**Structural Issue**) if all other columns agree and one deviates structurally.
- **Primary-input recomputation in parallel geography columns** (multi-geography CEAs only): When the primary column (typically column B or the first data column) contains a hardcoded value cited to an external paper or study, and adjacent geography columns (C, D, E, …) use formulas to independently re-derive what appears to be the same underlying quantity, flag as **Low/H** with Researcher judgment needed ✓ (**Structural Issue**): "Column B row [X] holds a hardcoded paper-derived value; columns C+ independently recompute the same quantity via formula. If all geographies share the same study finding, columns C+ should reference B[X] directly. If the recomputations are geography-specific, confirm they are not circularly derived from any model intermediate that itself traces back to B[X]." Do not flag when column B is also a formula — this check applies only when B is hardcoded from an external source and the parallel columns are formula-based.
- **Input row column-B reference collapse** (multi-geography CEAs only): For rows that serve as study-derived or model-input parameters in multi-geography models, scan whether all non-B geography columns are formula cells that simply reference column B (e.g., `=B21`, `=$B$21`, `=$B21`). If every non-B column on an input row resolves to the same column B value — making all geographies identical by reference rather than by independent estimate — and the column headers suggest geography-specific inputs are expected, flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**): "All geography columns on row [X] ('[label]') reference column B ([value]) rather than providing independent estimates. If this parameter is intentionally the same across all geographies (e.g., a global study finding that applies universally), add a cell note confirming this. If geography-specific values exist, update columns C+ with the appropriate values." This error is invisible when reading FORMATTED_VALUE mode because all cells show the same number — only FORMULA mode reveals the reference collapse. Read each cell in FORMULA mode before declaring this check clean. Check the existing `Absolute-reference note check` rule above: if a note already documents the uniform reference as intentional, do not flag.
- **Proportion/share distribution completeness**: When a section contains 3+ consecutive rows whose labels describe a partition of a total (e.g., "proportion of total efficacy provided by dose N," "share of deaths attributable to cause X," "weight assigned to component Y"), sum the values across all rows in the section and verify they total to 1.0 (or 100%). Trigger on row labels containing any of: "proportion of," "share of," "contribution of," "fraction of," "weight for," combined with a denominator concept ("total," "all doses," "all causes," "overall"). Flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**) if the sum deviates from 1.0 by >2 percentage points with no accompanying note explaining the gap (e.g., a residual category). Rationale: when a new component is added to a multi-row distribution block, existing proportions are rarely auto-renormalized, leaving the total silently above 100%.
- **SUMIFS/COUNTIFS staleness in Disease burden cells**: For key Disease burden cells that use SUMIFS or COUNTIFS to pull from large source data tabs (GBD, IHME, IGME), verify the formula result is consistent with a direct computation from the source data. Using the source tab data already read in the pre-read cache, independently compute the expected value by locating the relevant rows (matching year, geography, cause, and age group) and summing/averaging. If the independently computed value differs from the SUMIFS result by more than 1 percentage point, flag as **Low/H** (**Formula Error**): "SUMIFS result appears stale — formula returns X but direct read of [source tab] gives Y. SUMIFS formulas do not always recalculate automatically when the source data is updated." Scope this check to Disease burden cells whose formulas reference GBD, IGME, or IHME source tabs and whose output feeds a CE-chain parameter (mortality fraction, age distribution, or burden proportion). If the source tab is too large to verify from cache, note this and flag as Low/H with Researcher judgment needed ✓ asking the researcher to confirm the SUMIFS is returning the correct value against the current data.
- **Simple CEA vs. Main CEA final value cross-reference**: When the workbook contains both a Main CEA tab and a Simple CEA tab, identify the final CE value row on each tab (typically labeled "Cost-effectiveness after final adjustments" or similar). Compare the displayed value for each geography column across both tabs. If any geography column shows a discrepancy of more than 1% between Simple CEA and Main CEA, and no cell note or tab note explains the divergence, flag as **Medium/H** with Researcher judgment needed ✓ (**Inconsistency**): "Simple CEA [cell] = [value] but Main CEA [cell] = [value] for [geography] — [X]% divergence with no documented reason. If Simple CEA is intended to mirror Main CEA, trace which formula is incorrect." Recommended Fix: update the Simple CEA formula to reference the Main CEA final CE cell directly, or add a note explaining the intentional divergence. Skip this check if Simple CEA is designed as a standalone model with its own independent parameter inputs rather than as a cross-reference summary of Main CEA.

**Notes column accuracy — every row with a note, not just hardcoded cells**: For every cell that has a note — whether hardcoded or a formula — verify the note accurately describes what the cell actually contains. A note that misrepresents what the cell calculates, or cites a value that does not match the cell's current value, is a finding.
- **Note-formula inconsistency — determine fix direction before filing**: When a cell note and formula give conflicting values, do not assume the formula is wrong. Check for an adjacent changelog column, verify which direction is consistent with the model's logic, then recommend fixing the formula *or* the note.
- Source completeness (missing citations, first-person voice) is handled by the sources and readability agents — do not duplicate those checks here.

**Key parameter value cross-reference**: A dedicated **key-params-check** agent (Step 3e) performs the proactive scan for all GiveWell standard parameters — do not duplicate that work here. When the hardcoded value enumeration encounters a parameter that appears in `reference/key-parameters.md`, compare the stored value numerically against the entry in that file as a secondary safety net. A cell can cite the right source but store an outdated value — always compare the raw number, not the note. File any mismatch you encounter; severity and required output format are defined in `agents/key-params-check.md`.

**Asymmetric parameter updates across columns**: When a hardcoded value differs across columns representing parallel scenarios, verify the difference is documented. When you find an undocumented cross-column difference in a parameter with no structural reason to vary (a global adjustment factor, a moral weight, a program-level assumption), flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**).

**Percent vs. decimal scale consistency**: For any parameter row whose label describes a rate or proportion (mortality rate, coverage proportion, efficacy, prevalence, incidence, probability), compare values across all populated columns. Flag when two adjacent columns differ by approximately 100× — i.e., the ratio between them is between 90× and 110×. A column B value of `0.0047` alongside a column C value of `0.47` for the same labeled parameter row is a scale mismatch: one column expresses the quantity as a decimal, the other as a percentage. Flag as **High/D** (**Formula Error**): "[ref B] = [value] but [ref C] = [value] — a ~100× difference for the same parameter '[label]'. One cell likely expresses this as a decimal while the adjacent column uses a percentage. Apply `/100` or `*100` to whichever column has the wrong scale." Apply only to rows where the same underlying quantity should be consistent across columns; do not apply to rows where columns intentionally hold different quantities (scenario ranges, absolute counts vs. rates in the same section, or sensitivity bounds).

**Wrong-country note as value-error signal**: When a cell note references a different country than the column being analyzed, treat this as a trigger to verify the VALUE is correct — not merely a documentation issue. If the value is plausibly country-specific and no country-specific source exists, file as **High/H** with Researcher judgment needed ✓ (**Parameter Issue**).

**Stale-year note as value-error signal**: When a cell note cites a data vintage or source year that is more than 2 years before the model's grant period start year AND the row is a key epidemiological or cost parameter (mortality rate, incidence, coverage, unit cost, or any parameter in `reference/key-parameters.md`), treat this the same as a wrong-country note — verify the value itself, not just the documentation. A note reading "GBD 2019" in a 2025–2027 grant model is a trigger to check whether the underlying value has changed materially in the intervening years. If no updated value is available to confirm correctness, file as **Medium/H** with Researcher judgment needed ✓ (**Parameter Issue**): "Cell [ref] note cites [year] data for a key parameter in a [grant period] model — verify the value reflects the most recent available vintage and update the note." Do not file this finding if the note already explains why the older vintage is appropriate (e.g., "GBD 2019 used because the 2021 vintage does not disaggregate this age group").

**ITT vs. TOT internal consistency**: When a model uses an ITT effect estimate, verify it does NOT also apply a separate adjustment for absolute coverage — ITT already embeds partial take-up. When a model uses a TOT estimate, verify it DOES apply a coverage/take-up adjustment. Flag as **High/D** (**Formula Error**) if the ITT/TOT distinction is evident and the check fails. If the note does not indicate which was used, flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**).

**Aggregating-function as point estimate**: When a cell uses `AVERAGE`, `MIN`, `MAX`, or `MEDIAN` to compute a single best-guess input parameter — and neighboring cells in the same section are hardcoded single values — flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**). This signals unresolved uncertainty. Exception: if a cell note already explains the range average is intentional, do not flag.

Also apply this check within formula-only sections: when scanning a section where all rows use point-estimate formulas (single-cell references or arithmetic expressions) but one row uses `AVERAGE`, `MIN`, or `MAX` without a cell note explaining why, flag as **Low/H** with Researcher judgment needed ✓ (**Assumption Issue**): "[cell] uses AVERAGE() while all adjacent rows in this section use point estimates — add a note explaining why an average is appropriate here and which values are being averaged." This catches the case where AVERAGE is inconsistently applied to one metric within a uniform section, which typically signals either an unresolved choice between data vintages or an undocumented methodological decision.

**Within-section formula homogeneity check**: When reading cells in the same column within a logical section, flag any cell that uses a formula while cells immediately above and below it are hardcoded constants — or vice versa. File as **Low/H** with Researcher judgment needed ✓ (**Structural Issue**). Exception: do not flag where a cell note documents why the derivation method differs.

**Annuity-due vs. annuity-immediate** (mandatory for every `PV()` formula): Inspect the `type` argument. `type=0` is annuity-immediate (standard). `type=1` is annuity-due, which overstates PV by approximately `(1 + r)`. For every `PV()` formula, if `type=1` and no cell note explains why beginning-of-period is appropriate, file as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**).

**Hardcoded numeric literals embedded inside formulas**: When enumerating hardcoded cells, also scan for numeric literals hardcoded *within* formulas — e.g., `=2.47%*C43`. These are NOT captured by the Hardcoded Values agent (which reads only non-formula cells). Flag any such embedded literal as "Missing Source" in Publication Readiness if it drives a meaningful input (prevalence rate, adjustment factor, conversion parameter) and no cell note cites its origin. Note: do NOT route missing-source findings for *standalone* hardcoded cells (plain values not inside formulas) to Publication Readiness — those are already tracked in the Hardcoded Values sheet (column F).

**SUMPRODUCT array constant audit**: For any SUMPRODUCT formula containing an array constant — e.g., `=SUMPRODUCT({0.8,0.9,0.7},{C14:C16})` or `=SUMPRODUCT({$H34,$H34,$H32},{'Source tab'!L65:'Source tab'!L67})` — flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**) if the array drives a CE-chain calculation and no cell note documents the source and meaning of each element. Array constants are effectively undocumented hardcoded values: they lack row labels, won't update if model inputs change, and cannot be traced to a source tab. Recommended Fix: replace the array constant with a named range or cell references pointing to a documented parameter block, and add a note citing the source for each element.

**INDIRECT and OFFSET formulas**: For any cell containing `INDIRECT()` or `OFFSET()`, flag as **Low/O**: "Formula uses INDIRECT/OFFSET, which creates a dynamic reference that cannot be audited by formula inspection alone — the referenced cell depends on runtime values. Verify manually that the dynamic reference always resolves to the intended target under all scenario inputs."

**Hidden rows and columns**: When reading sheet data, check for gaps in the row sequence returned (rows that appear skipped in the batch response). If rows appear missing (e.g., data jumps from apparent row 15 to row 17), this may indicate hidden rows containing data not visible in the normal view. Flag as **Low/O**: "Row numbers [X] through [Y] appear hidden in [sheet name]. Hidden rows may contain parameter values or formula intermediates not visible in the published view. Verify that no active calculations or assumptions are stored in hidden rows."

**Relative Risk ratio direction**: When a formula computes `=(RR_A / RR_B) × other_terms` to derive a relative benefit adjustment, flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**): does this intend to compare the RRs themselves, or the mortality reductions `(1-RR_A)/(1-RR_B)`? These give the same answer only when `RR_A ≈ RR_B`. Confirm which is intended in a cell note.

**INDEX/MATCH lookup key verification**: For any formula using `INDEX(..., MATCH(key, lookup_range, match_type))` to navigate a source tab, verify: (a) the `match_type` argument — when `match_type` is `1`, `-1`, or omitted (the default is `1`, approximate match ascending), the function silently returns the nearest value rather than an exact match, which has the same wrong-row failure mode as VLOOKUP approximate match; when `match_type` is `0`, proceed to key verification; (b) for exact-match (`match_type=0`) formulas, verify the MATCH key — whether a hardcoded string or a cell reference — exactly matches the intended identifier in the lookup range. A key of `"Kaduna"` in a range containing `"Kaduna - Cohort 17"` will resolve to the wrong row or return `#N/A`. For hardcoded string keys, read the lookup range and confirm the exact string appears. For cell-reference keys, confirm the referenced cell's value matches the corresponding column or row header in the source tab. Flag as **High/D** (**Formula Error**) if the key doesn't match the intended entry: "[cell] uses MATCH('[key]') but the source lookup range contains '[actual label]' — the lookup resolves to the wrong row/column." For approximate-match MATCH (match_type omitted, 1, or -1), flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses MATCH with match_type=[value/omitted] — approximate match. This requires the lookup range to be sorted ascending/descending and silently returns the nearest value on unsorted data. Change to match_type=0 for exact match, or confirm the lookup range is sorted and approximate match is intentional."

**VLOOKUP approximate-match audit**: For any formula using `VLOOKUP(key, range, col_index)`, inspect the fourth argument (`range_lookup`). When the argument is `TRUE`, `1`, or omitted (approximate match is the default in Google Sheets), the function requires the lookup column to be sorted ascending — on unsorted data it silently returns the wrong row with no error indicator. For every VLOOKUP in the workbook: if the fourth argument is missing or `TRUE`, flag as **Medium/H** with Researcher judgment needed ✓ (**Formula Error**): "[cell] uses VLOOKUP with approximate match (`range_lookup` omitted or TRUE) — this returns the nearest-lower value, not an exact match. On unsorted lookup data this silently returns the wrong row. Change to FALSE for exact match, or confirm the lookup column is sorted and approximate match is intentional." If the fourth argument is `FALSE` or `0`, no finding.

**YEAR() applied to year-value check**: For any formula containing `YEAR()`, inspect the argument. If the argument resolves to or contains an integer in the range 1900–2100, Google Sheets interprets that integer as a serial date number rather than a calendar year — e.g., `YEAR(ROUND(J284,0))` where J284 = 2027 returns 1905, not 2027 (since day 2027 from Dec 30, 1899 falls in 1905). This produces a silently wrong result with no error indicator. Flag as **Low/H** (**Formula Error**): "[cell] applies YEAR() to a value of [X] — Google Sheets treats integers as serial dates, so YEAR([X]) returns [wrong year], not [X]. Remove the YEAR() wrapper and reference the cell directly." This pattern most often appears in trend or projection rows where a year value is stored in a cell and then wrapped in YEAR() to extract it — a no-op that corrupts the value.

**IFERROR / IFNA error-suppression audit**: For any formula of the form `=IFERROR(formula, fallback)` or `=IFNA(formula, fallback)` whose result feeds a CE-chain calculation — i.e., the cell's value is referenced downstream in the cost or benefit chain — flag as **Low/H** with Researcher judgment needed ✓ (**Formula Error**) when the fallback value is `0`, a blank, or any numeric literal: "[cell] uses IFERROR/IFNA with a numeric fallback of [value]. If the inner formula fails — e.g., due to a broken cross-sheet reference, a missing row, or a lookup miss — the cell silently returns [value] rather than surfacing an error. Confirm the inner formula cannot legitimately fail under current inputs, or replace the fallback with a string like 'ERROR — check source data' so failures are visible." Exception: do not flag `IFERROR(..., "")` or `IFERROR(..., "N/A")` used in display-only cells that are not part of the CE chain.

**Structured source-tab reference verification**: For every formula that pulls data from a structured source tab — where rows represent a concept (disease, vaccine dose, age group, efficacy estimate) and columns represent geographies — verify both (a) the row label at the referenced cell describes the correct concept for this formula, and (b) the column header at the referenced cell matches the geography the formula is computing for. This applies to all-cause mortality rates, disease burden, population figures, efficacy/treatment effect data, and coverage data pulled from source tabs. A formula computing measles-dose-1 outcomes that references a rotavirus row, or a formula computing Niger State CE that references the Kano column of a GBD tab, is a **High/D** (**Formula Error**): "[cell] references [source tab]![cell ref] (row label: '[label]', column header: '[header]') but this formula computes [intended concept] for [intended geography] — correct to reference [correct cell]." Produce a verification table for every structured source-tab cross-sheet reference:

| Formula cell | Row geography / concept | Source tab cell ref | Referenced row label | Referenced col header | Row match? | Col match? |
|---|---|---|---|---|---|---|
| [ref] | [geography/concept] | [cell] | [row label] | [col header] | YES/NO | YES/NO |

A row absent from this table has not been checked. Coverage declaration: "Structured source-tab reference check complete. Cells checked: [N]. Issues found at: [list or 'none']."

**Sibling-range consistency after a range-endpoint flag**: Whenever a range-endpoint miss is filed — e.g., a SUMPRODUCT or SUM using rows 35:36 when row 37 also belongs to the same conceptual block — immediately scan ALL other formulas in the same sheet that reference that source row range for the same truncation. A range-endpoint miss in one formula is almost always replicated in parallel formulas (AVERAGE, SUM, SUMPRODUCT) covering the same conceptual block in adjacent rows, because they were built together by copy-paste. Do not stop after filing the first miss — scan every formula in the section for the same pattern before moving on. Flag each sibling miss as a separate **Medium/H** (**Formula Error**) finding: "[cell] uses [formula] — same truncated range as [original finding cell], omitting row [N] ('[label]') from the [section] calculation."

**Important**: this sibling scan must extend **beyond your assigned row scope** (`{scope_start}` to `{scope_end}`). Read at least 30 rows above and below the flagged cell on the full sheet — sibling formulas covering the same conceptual block frequently fall in an adjacent parallel instance's assigned range and will be missed if the scan stops at the instance boundary. File any sibling misses found outside your scope as normal findings; the compaction agent will sort them correctly.

**Multi-geography column structure checks** (multi-country CEAs only): When a workbook models multiple geographies in parallel columns:

- **Shared-vs-geography-specific conflict**: For every parameter row, verify the value is populated in *either* the shared column *or* the geography-specific columns — not both. A value appearing in both creates an ambiguity about which one the model actually uses. Flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**).
- **Unfunded geography convention**: When a geography column is currently unfunded/inactive, verify it uses `0.01` (not `0`) as the placeholder. Using `0` in a denominator position will cause `#DIV/0!` errors or silently zero out CE. Flag any `0` in an unfunded geography's cost or coverage row as **Medium/H** (**Formula Error**).

**New geography column audit** (triggered when the user declares new geography columns were added): Treat every formula in those new columns as a mandatory cell-level check. Do not sample. For each new column, verify:

1. **GBD location reference**: For each formula referencing a GBD or disease burden tab, confirm the row index maps to the correct location. Read the GBD tab at the exact row index used in the formula and check its location label. If the location name does not match the column header, flag as **High/D** (**Formula Error**).
2. **GBD age group**: For each GBD-sourced cell, verify the referenced row uses the correct age group — not "All ages" when the model requires an age-specific estimate (e.g., under-5). A label mismatch is **High/D** (**Formula Error**) regardless of how close the values appear.
3. **Locked column references**: Scan every formula in the new columns for locked column references (`$B$xxx`). When the locked column is the national column and the referenced row could have geography-specific values, flag as **Medium/H** with Researcher judgment needed ✓ (**Assumption Issue**). **Before filing**: read the column A cell note for that row. If the note explains the absolute reference is intentional (e.g., "all geographies use this reference because the study was conducted in [country]"), do not file.
4. **Explicit zeros for geography-scoped parameters**: When a parameter row is labeled as applying to a subset of geographies only, verify all non-applicable columns have explicit zero values rather than blank cells.

**Before writing any findings from this section**, produce an audit log for every new column × every check: `Column C: (1) GBD location ✅ / (2) GBD age group ❌ / (3) locked refs ✅ / (4) explicit zeros ✅`. This log is a completion gate — do not skip it.

**Parallel-column formula coverage completeness** (any model with parallel formula columns): When a workbook models multiple geographies, cohorts, or scenarios in parallel columns, identify the rightmost populated data column — this is the last column the model covers. Then for every formula section that aggregates across columns (benefit calculations, EV adjustments, cost summaries, outcomes totals), verify that every SUM/SUMPRODUCT/AVERAGE formula range extends to that last column. A formula range that stops at column N−1 when column N exists is a **High/D** (**Formula Error**): "[Row ref] formula sums only through [last column in formula] — column [X] ([geography/cohort/scenario]) exists but is excluded from the range. All downstream CE calculations that depend on this row omit that column's data." Recommended Fix: extend the formula range to include all populated columns, or add a note confirming the omission is intentional (e.g., data is not yet finalized). This check applies specifically to rows computing aggregate totals — not to rows that are intentionally column-specific. After completing this check, write: "Parallel-column coverage check complete. Last populated column: [X]. Rows with truncated ranges: [list or 'none']."

**Source-tab row-range completeness** (multi-disease/multi-vaccine models): When a formula aggregates across a consecutive block of rows in a source tab — e.g., `SUM('Disease burden'!B446:B457)` spanning all vaccine doses — verify the row range covers all defined conceptual inputs. Identify the full set of rows the source tab defines for that concept (e.g., one row per vaccine dose or disease), then check the formula's range endpoints against that full set. A range stopping one row short silently excludes the last disease or vaccine from the calculation. Flag as **High/D** (**Formula Error**): "[cell] aggregates [source tab] rows [X]–[Y] but the source tab defines [concept] through row [Z] — row [Z] ('[label]') is excluded from the range." After completing this check, write: "Source-tab row-range completeness check complete. Aggregation formulas checked: [N]. Truncated ranges found at: [list or 'none']."

**Parallel-column row-reference consistency** (any model with parallel formula columns): For every section that computes structurally parallel formulas across columns representing distinct geographies, cohorts, or scenarios — EV adjustments, benefit calculations, efficacy inputs — verify each column's formula references the column-appropriate row. The common failure mode: a copy-paste error where one column's formula was not updated, leaving it referencing the same source row as another column (e.g., Cohort 9's EV formula still references the "Cohort 1–8" row, or a country column referencing another country's disease burden row).

For every parallel-column section, read the formula in each column and record the referenced row's label. Produce a verification table:

| Section | Row label | Column 1 ref → row label | Column 2 ref → row label | … | Pattern consistent? |
|---|---|---|---|---|---|
| EV adjustments | [label] | [ref] → [label] | [ref] → [label] | … | YES/NO |

If any column's formula references a row labeled for a different geography, cohort, or scenario (or a pooled/mean row when column-specific rows exist), flag as **High/D** (**Formula Error**): "[cell] in the [section] section references '[stale row label]' — this is [Column X]'s row, not [Column N]'s. All [section] calculations for [Column N] use [Column X] data." A row absent from the table has not been checked. Coverage declaration: "Parallel-column row-reference check complete. Sections checked: [list]. Issues found at: [list or 'none']."

## Writing Findings

**Severity guard for value-error claims**: Before filing a finding that classifies a specific hardcoded value as *wrong* (High/D), you must have done at least one of: (a) read the cell's linked source document and confirmed the value contradicts it, (b) verified against `reference/key-parameters.md`, or (c) confirmed via arithmetic that the value is internally inconsistent. Do not file a value-error High/D based solely on reasoning. If uncertain, downgrade to Medium/H with Researcher judgment needed ✓.

Before writing any finding, confirm: (1) the exact cell reference(s) affected, (2) the specific value or formula that is wrong, (3) the precise fix required.

**Your row start position is pre-assigned in session context** — do not auto-detect. Append findings using `modify_sheet_values`. See `reference/column-reference.md` for full column specifications.

Column reference: **A** Finding # (leave blank) | **B** Sheet | **C** Cell/Row | **D** Severity | **E** Error Type/Issue (write the exact label only — no additional text, description, dashes, or punctuation after it; choose one of: Formula Error | Parameter Issue | Adjustment Issue | Assumption Issue | Structural Issue | Inconsistency) | **F** Explanation (1–2 sentences max; lead with the specific problem; make a specific falsifiable claim and include the actual value or formula, e.g., "B14 = 0.87 but C22 = 0.79"; plain language; do not hedge what you can confirm; no chain traces) | **G** Recommended Fix (one sentence or formula only; lead with an imperative verb; include the exact replacement formula or value; no explanation of why) | **H** Estimated CE Impact (write exactly one of these standard phrases — no other wording: Raises CE — [estimate] | Lowers CE — [estimate] | Raises CE — magnitude unknown | Lowers CE — magnitude unknown | No CE impact | Direction unknown; for Raises CE and Lowers CE, replace [estimate] with the actual CE multiple, e.g., Raises CE — 8.7x → ~10.2x) | **I** Researcher judgment needed (✓ only for intent/decision questions — not for "please verify" tasks) | **J** Status (leave blank)

**Overflow protection**: If you exhaust your allocated row budget and still have findings to write, do not stop. Continue writing at the next row beyond your budget — the compaction agent reads all rows and will sort any overflow findings into their correct position.

Group findings where the same issue applies to multiple cells — aim for ~15–25 grouped findings, not one row per cell.

**Publication Readiness column layout differs**: When routing a finding to Publication Readiness, use the 6-column A–F layout. Write exactly 6 values per row — no more. Do not include Severity, Status, Changes CE?, Estimated CE Impact, or Researcher judgment needed. Writing a 7th column will corrupt the sheet layout. A=Finding # (blank) | B=Sheet | C=Cell/Row | D=Error Type/Issue (write the exact label only — no additional text, description, dashes, or punctuation after it; choose one of: Missing Source | Broken Link | Permission Issue | Readability | Terminology) | E=Explanation | F=Recommended Fix.

After all findings are written and all other steps are complete, write ONE final row to the Findings sheet at the next available row within your allocated range (or at the first row of your allocated range if no findings were written). This is the absolute last action you take before finishing.

Write the row with:
